# Authority And Memory Spec

## Purpose

This spec defines how MyLittleHarness separates product authority, operating memory, compatibility fixtures, and future generated views.

## Authority Model

MyLittleHarness authority is file-first. Accepted decisions, current focus, plan status, stable rules, and carry-forward fates must be recoverable from repo-visible artifacts.

Product docs under `docs/...` describe reusable architecture and product contracts. They do not operate the current workflow by themselves.

The target repository owns active focus, current plan status, research intake, and closeout evidence for the repository MyLittleHarness is servicing. The product repository owns reusable source, tests, and product documentation. Legacy reference material is opened only for a named blocker.

Live-root route discovery is reportable helper context, not a new authority layer. Root kind is a coarse routing posture, not proof that lifecycle frontmatter is valid or that a command may mutate. A `workflow-core` manifest plus malformed or partial project-state frontmatter does not make a root eligible as a live operating root; route files and command-specific authority still need direct validation. `status`, `check`, and `intelligence --focus routes` may print a compact lifecycle route table for live operating roots so agents can place state, active plans, optional roadmap sequencing, incubation, research, specs, decision/do-not-revisit records, ADR records, verification through the active-plan block and optional `project/verification/*.md` proof/evidence records, closeout/writeback, archives, and docs routing without growing `AGENTS.md` into a dense manual. `intelligence --focus routes` also reports deterministic present-surface classification from the product route registry, mapping discovered repo-visible files to routes such as state, active-plan, roadmap, incubation, research, stable-specs, docs-routing, product-docs, archive, decisions, ADRs, verification, generated-cache, or operating-guardrails. `project/roadmap.md` may sequence accepted work between incubation and one active implementation plan, but it is not part of the mandatory start path and cannot approve repair, closeout, archive, commit, rollback, or lifecycle decisions. `status` and `check` also summarize first-class lifecycle pointers so `phase_status = complete` is visible as "implementation work is not pending" and ready for explicit closeout/writeback without approving archive, roadmap done-status, next-slice movement, or manual commit. Product-source fixtures must not present that table as live memory, must not present discovered surface inventory as live memory, and must not present the continuation summary as live memory; route output cannot approve mutation, repair, closeout, archive, commit, or lifecycle decisions.

Read-only route metadata validation is also live-root-only. `validate`, `check`, and `check --focus validation` may warn about malformed lifecycle frontmatter relationship fields on routed roadmap, research, incubation, stable-spec, decision, ADR, and verification artifacts, including unsafe root-relative paths, missing archive targets, stale active relationship references, and destination-route mismatches. Strict fields such as `promoted_to` and `archived_to` remain path-only; flexible relationship fields may carry prose labels, but only path-like values are validated as local targets. Roadmap relationship fields such as `related_roadmap`, `source_roadmap`, and `archived_plan` are path-checked when present; the separate hidden `roadmap --dry-run|--apply` rail is the only implemented roadmap write path. Plan-facing stable specs and posture-opted-in `docs/specs/**` product specs may also carry separate `spec_status` and `implementation_posture` metadata, where target-only means accepted contract not yet matched by implementation and synced means source-bound verification exists. Spec posture findings such as missing posture, synced-without-verification, target-only-with-evidence, drift-without-carry-forward, or superseded-without-target are read-only reconcile cues. `check --focus archive-context` is a separate read-only audit over `project/roadmap.md`, `project/archive/plans/*.md`, and source incubation/research references; it reports archive coverage counts, missing archived-plan target root-cause classes with route traces, stale source references, reconstructed source evidence, suspect incomplete-context archives, and bounded recovery actions. These `route-metadata-*`, `spec-*`, and `archive-context-*` findings are advisory diagnostics; they do not create a repair class, approve spec deletion, approve archive movement, or authorize closeout, commit, rollback, or lifecycle decisions.

## Memory Rule

An operating root has one mutable project memory surface for active focus and canonical pointers.

The product tree must not introduce a second mutable memory tree. The fixture `project/project-state.md` inside a product source checkout is compatibility data for CLI/tests, not operating memory.

The final product model is MyLittleHarness attached to a target repository, with accepted project truth recoverable from that repository's visible files.

## V2 Agent Governance Foundation

The v2 architecture direction keeps the same product model and makes it more legible to external agents. MyLittleHarness v2 is the deterministic State around a target repository, not the agent runtime itself. External humans, agents, model providers, MCP servers, search indexes, CI systems, issue trackers, and notification relays are clients or embassies around the State; they can help, but they do not become sovereign authority.

The v2 foundation is a protocol surface:

- route manifest and structured findings describe what routes exist, which routes are authority, which outputs are generated maps, and which transitions require a human gate;
- role profiles describe allowed reads, proposals, apply rails, forbidden actions, context tiers, required outputs, and gate classes;
- agent run evidence records who did what, from which inputs, with which outputs, commands, source hashes, stop reasons, and optional model/tool provenance;
- handoff packets and work claims make fan-out explicit before parallel workers touch overlapping paths;
- run contracts and receipts bind a single target root, role, queue item, handoff refs, claim refs, planned agent-run refs, allowed routes, write/review scope, required outputs, fan-in checklist, and route proposals without becoming a scheduler;
- session-scoped active work records show each agent session or run's current plan, phase, slice, heartbeat, and lease posture without rewriting the global project-state active plan pointer;
- review tokens and approval packets bind human approval to planned writes, source hashes, gate class, allowed decisions, and verification evidence;
- reconcile/drift records compare intended law with observed code/evidence and propose amendments or drift decisions without rewriting accepted truth.

These surfaces do not imply `swarm run`, hidden queues, provider credentials, model gateway behavior, daemon locks, automatic worker spawning, autonomous release, or direct worker writes to shared lifecycle authority. The first v2 product work should document and expose route/role/evidence law before any runtime orchestration exists.

The multi-agent direction is coordination-first: repo-visible routes stay the durable authority while adapters make current work easier to see and steer. The accepted contract says claims, agent-run records, handoff packets, and session-scoped active work are the first-class coordination layer; they show who owns scope, what was assigned, what was done, and what remains recoverable without chat memory. The accepted contract says hooks are sensors, blockers, and context injectors around those records, not lifecycle owners. The accepted contract says dashboard and `mlhd` runtime surfaces are cockpit projections over repo-visible routes and disposable runtime observations. A dispatcher preflight cannot report launch-ready until a handoff packet, active non-stale claim, planned or recorded agent-run path, ready repo-visible queue item, terminal current upstream blockers, non-conflicting write/review scope, and non-lifecycle allowed routes are all clean. Even then the surface is a read-only gate around explicit MLH rails, not an authority surface and not a worker launcher.

An optional external orchestrator may own the runtime packet for roles, pipeline ordering, queue dependencies, retries, backend choice, provider state, and worker invocation. MyLittleHarness owns the repo-visible run contract and receipt that an orchestrator must satisfy before and after a worker run. That split matches the external agent runtime model: an SDK or orchestration layer may manage turns, tools, handoffs, guardrails, results, tracing, and resumable state, while application policy decides which tools, roots, routes, write scopes, and approvals are allowed. MLH validates those policy invariants from files; it does not become a provider gateway or a second scheduler.

Runtime state must use a separate vocabulary from lifecycle authority. The first worker receipt namespace is `runtime_state_namespace.v1`; it keeps `runtime_status`, `worker_status`, `workflow_status`, `verification_verdict`, `lifecycle_status`, and `research_import_status` distinct so an SDK worker can report that it ran or succeeded without implying that MLH accepted the work, moved a roadmap item, archived a plan, or approved provider/Git/release actions. Worker receipts may cite those namespace values as evidence, but the lifecycle transition still belongs to explicit MLH dry-run/apply routes and repo-visible closeout facts.

Worker event histories are receipt provenance, not hidden memory authority. A worker receipt may summarize repo-visible event streams and record whether private SDK traces were redacted, summarized, excluded, not recorded, or unknown, but any private trace policy must say that private traces are non-authoritative and recovery depends on repo-visible evidence. Event-history summaries and trace metadata cannot accept lifecycle state, fan-in work, roadmap done-state, archive, Git, provider routing, release, or cleanup.

Runtime guard preflight receipts are nested worker receipt evidence, not launch authority. A worker receipt may include `runtime_guard_preflight` using schema `mylittleharness.runtime-guard-preflight-receipt.v1` to record mission preconditions, runtime readiness, replay posture, worktree clean/dirty state, unsafe bypass flags, dispatch/mailbox refs, leader-owned lifecycle refs, native-hook or fallback proof, and provider proof. These fields are diagnostics for an optional external runtime to decide what evidence exists before and after a worker attempt. They cannot launch workers, approve route proposals, accept fan-in, choose providers, approve lifecycle movement, stage, commit, push, archive, release, clean up, or accept a target repository. Dirty worktree posture and unsafe bypass flags are allowed as evidence only when they do not claim permission, and fallback/synthetic hook observations must not be described as native hook proof.

Checkpoint/resume receipts are nested worker receipt evidence, not durable execution authority. A worker receipt may include `checkpoint_resume` using schema `mylittleharness.checkpoint-resume-receipt.v1` to record durable task/run ids, trace/span refs, idempotency posture, checkpoint kind/status, resume requirement/status/reason, restore strategy, replay refs, checkpoint failure status, and backpressure verdict posture. These fields let an optional external runtime explain resumability and queue/backpressure state without claiming MLH accepted work. Queue success, run completion, replay success, checkpoint creation, stale backpressure verdicts, websocket connection, and customer-owned message hydration cannot approve lifecycle movement, fan-in, route proposals, verification acceptance, worker launch, roadmap status, archive, Git, provider routing, release, cleanup, or target-repo acceptance.

Child agent fanout receipts are nested worker receipt evidence, not orchestration authority. A worker receipt may include `child_agent_fanout` using schema `mylittleharness.child-agent-fanout-receipt.v1` to record a parent run id, fanout status, child worker list, coordination refs, runtime posture refs, task/prompt/output/event-history/verification refs, namespace-separated child statuses, and residual risk. This lets an optional SDK runtime leave repo-visible proof that parent/child worker coordination occurred. It does not approve fan-in, accept child outputs, move lifecycle state, mark roadmap work done, approve route proposals, choose providers, stage, commit, push, archive, release, clean up, launch more workers, or accept a target repository. Child worker success, reviewer approval, worktree posture, bypass posture, hook proof, provider proof, and private traces remain evidence only.

Capability fence decision receipts are nested worker receipt evidence, not tool authorization authority for MLH lifecycle. A worker receipt may include `capability_fence_decision` using schema `mylittleharness.capability-fence-decision-receipt.v1` to record a fence id, capability profile, capability-fence status, optional approval/audit status, policy refs, tool manifest refs, claim/handoff/session refs, gateway refs, and requested/allowed/denied capability lists. These fields let an optional SDK runtime explain which tool or capability policy decision was observed before or during a worker attempt. They cannot approve lifecycle movement, worker launch, fan-in, verification acceptance, roadmap status, archive, Git, provider routing, release, cleanup, route proposals, or target-repo acceptance. Allow decisions, denial decisions, approval states, audit records, policy hashes, and gateway outcomes remain source-bound evidence only.

Runtime broker/provider receipts are nested worker receipt evidence, not provider routing, launch, credential, workspace, or cleanup authority. A worker receipt may include `runtime_broker_provider` using schema `mylittleharness.runtime-broker-provider-receipt.v1` to record runtime broker id/provider id, broker/provider/registration/join/server/dispatch status, workspace isolation and cleanup posture, credential projection posture, observed approval mode, resume posture, telemetry posture, refs, and nested broker/provider/workspace/credential/cleanup/resume/telemetry evidence. These fields let an optional SDK runtime explain where a worker was eligible to execute and what runtime boundary evidence exists. They cannot approve lifecycle movement, worker launch, tool execution, provider routing, fan-in, verification acceptance, roadmap status, archive, Git, release, workspace cleanup, credential projection, telemetry retention, route proposals, or target-repo acceptance. Broker dispatch, default-provider selection, withdrawn/orphaned workers, workspace mounts, path guards, cleanup results, credential projection, harness full-auto modes, resume/relaunch events, telemetry exports, worker success, reviewer approval, and verifier pass remain source-bound evidence only.

Worker worktree/session receipts are nested worker receipt evidence, not launch, merge, cleanup, Git, verification, or lifecycle authority. A worker receipt may include `worker_worktree_session` using schema `mylittleharness.worker-worktree-session-receipt.v1` to record session id, worktree/session status, worktree and branch refs, prompt and prompt-file refs, status and capture refs, session/pane refs, sandbox refs, merge and cleanup refs, wait refs, concurrency refs, dashboard refs, and nested worktree/prompt/session/status_capture/sandbox/merge_cleanup/concurrency evidence. These fields let an optional SDK runtime explain which worktree, terminal pane, prompt, capture, sandbox posture, wait result, merge result, or cleanup result was observed. They cannot approve lifecycle movement, worker launch, fan-in, verification acceptance, roadmap status, archive, Git, provider routing, release, cleanup, route proposals, or target-repo acceptance. Terminal panes, status dashboards, wait success, sandbox declarations, merge success, cleanup success, worker success, reviewer approval, and verifier pass remain source-bound evidence only.

Artifact lineage receipts are nested worker receipt evidence, not artifact acceptance, hash-chain authority, signature/HMAC authority, or lifecycle authority. A worker receipt may include `artifact_lineage` using schema `mylittleharness.artifact-lineage-receipt.v1` to record lineage id/status, output/input/parent/prompt/verification/signature refs, producer worker/run ids, prompt/model/provider/cost/token metadata, content and parent hash posture, signature/HMAC posture, and lineage verification status. These fields let an optional SDK runtime explain where an output came from and what verification evidence exists. They cannot approve lifecycle movement, verification acceptance, artifact acceptance, target-repo acceptance, fan-in, roadmap status, archive, Git, provider routing, release, cleanup, route proposals, or worker launch. Lineage records, content hashes, parent hashes, signatures, HMAC chains, prompt/model/provider/cost/token metadata, worker success, reviewer approval, and verifier pass remain source-bound evidence only.

The task-session surface makes that runtime split explicit. `task-session --inspect --json` is a read-only `mylittleharness.task-session.inspect.v1` preflight packet for external runtimes. It projects lifecycle, active plan, roadmap queue, topology, coordination posture, capability flags, and the next safe navigation command from repo-visible files. Its readiness fields, including `external_runtime_may_launch_workers`, are scheduling hints for the external runtime only; they do not start workers, approve lifecycle movement, accept route proposals, approve fan-in, choose providers, stage Git, archive, close out, or mark roadmap work done.

`task-session --inspect --fan-in --json` is the corresponding read-only fan-in readiness packet, with schema `mylittleharness.task-session.fan-in.inspect.v1`. It lets an optional external runtime ask MLH whether the active session graph has enough repo-visible evidence to be considered ready for a later reviewed fan-in step. The answer is evidence only: `ready` means MLH can see the needed work-claim, handoff, agent-run, route-proposal, and matching task-session receipt refs for the active session or execution slice; `blocked` names missing or stale graph pieces; `not-required` means the fan-in gate is not activated; `refused` means the root posture is not eligible. None of those states release claims, accept route proposals, launch workers, choose providers, approve lifecycle movement, write back, archive, stage, commit, push, mark roadmap work done, or enable cross-root execution.

External completion claims are report-only unless they are converted into MLH repo-visible evidence. `claim --action release` requires an explicit release condition and records a `completion_policy` object plus `completion_evidence` on the work-claim record. That policy is `repo-visible-evidence-only`: external tracker/orchestrator status, comments, queue state, SDK traces, or runtime memory may explain why a worker stopped, but they are not authoritative done-state and cannot approve closeout, roadmap done-status, archive, staging, commit, push, or Git. `claim --status`, `check --focus agents`, and `task-session --inspect --conductor --json` may surface the policy for operators and external runtimes, but the policy still grants no external tracker mutation and no lifecycle authority.

`task-session --dry-run|--apply` is the bounded receipt rail for that preflight contract. Apply may create exactly one JSON receipt under `project/verification/task-sessions/*.json` using schema `mylittleharness.task-session.receipt.v1`. The receipt records runtime owner/backend, task and session ids, objective, execution slice, read context, write scope, allowed routes, stop conditions, required outputs, evidence refs, claim refs, handoff refs, route proposal refs, provider refs, dirty-start VCS posture, and the current preflight snapshot. It also records reserved topology fields such as `root_id`, `coordination_root`, `target_roots`, `evidence_owner`, `fan_in_owner`, `queue_owner`, and `cross_root_allowed_routes`. Those fields are contract data for later multi-root work, not permission to give workers multiple roots or execute cross-root routes.

Task-session receipts are evidence, not authority. Receipt approvals for lifecycle, fan-in, route proposal, provider routing, and Git remain false. Receipt validation refuses lifecycle-authority write scopes and forbidden routes such as plan, roadmap, repair, writeback, transition, archive-shaped movement, Git, provider configuration, hooks, or cleanup. Private runtime traces, provider state, queue state, and SDK telemetry may help diagnose a run, but a task is recoverable only when the durable result is written back as MLH-compatible repo-visible evidence.

Bounded SDK tool profiles stay outside MLH runtime control, but MLH can validate their recorded invariants. Researcher and verifier profiles should be limited to read/search/source inspection and current queue update evidence. Implementer profiles may have scoped writes only inside a declared claim. No worker-facing profile may carry lifecycle apply rails, repair apply rails, roadmap apply rails, writeback apply rails, Git mutation, provider configuration, archive movement, or release authority unless a future explicit human-gated contract says otherwise.

The `session-start` hook event is a projection of that same authority model into first-contact client context. Its structured hook-event payload may carry lifecycle posture, next legal dry-run guidance, dashboard agent packet data, and projection/SQLite cache posture, but the payload is not memory authority and not completion evidence. It cannot approve lifecycle movement, product-diff scope, Git actions, dispatcher launch, provider routing, generated-cache truth, archive, roadmap status, staging, commit, push, or release. When hook output conflicts with route files, the route files and explicit MLH dry-run/apply rails win.

Session-scoped active work lives under `project/verification/session-active-work/*.json` as read-only coordination evidence. `check` reports it separately from the global lifecycle summary so parallel agents can see active, stale, expired, and unknown session states without treating a session record as the active plan pointer. These records are not closeout facts, not locks, and not approval to archive, mark a roadmap item done, stage, commit, roll back, release, or open the next slice; `project/project-state.md` frontmatter remains the global lifecycle authority.

The implemented role-profile slice is read-only protocol data. `manifest --inspect --json` includes `role_manifest` entries for advisory offices such as planner, coder, verifier, archivist, and governor. Each entry carries route permissions, forbidden actions, required outputs, context/output packet requirements, gate classes, and human gates. No role profile carries direct apply authority; explicit MLH dry-run/apply rails and reviewed human gates remain the only lifecycle mutation path.

Autonomous audit context stays tiered so the operator can inspect a root without turning every available file or report into hot prompt state:

- Hot Authority: `AGENTS.md`, `.mylittleharness/project-workflow.toml`, `project/project-state.md`, the active plan when `plan_status = "active"`, accepted stable specs or decisions needed for the task, and current verification or closeout facts. Legacy `.codex/project-workflow.toml` is a fallback only when the neutral manifest is absent. These files are the default recovery surface and cannot be replaced by generated summaries.
- On-Demand Routes: `project/roadmap.md`, same-topic incubation or research, docs routing, route manifests, focused `check`, `intelligence`, `memory-hygiene`, and route-reference reports. These are read when the current question needs their lane; they route attention but do not approve writes.
- Generated Projections: `.mylittleharness/generated/projection/**`, link/backlink/fan-in/relationship summaries, SQLite search indexes, generated manifests, and other rebuildable navigation caches. They can speed audit navigation and stale-reference discovery, but deletion or staleness must not change accepted truth.
- Archive/Evidence: archived plans, archived reference material, `project/verification/*.md`, agent-run records, work claims, handoff packets, session-scoped active work records, approval packets, review-token inputs, and durable proof/evidence records. These are cold source-bound evidence recovered by pointer for audit, calibration, or closeout support; they are not default continuation state and cannot approve lifecycle movement by themselves.

Selective forgetting is compaction and archive with provenance. It is not deletion of authority and not generated-memory sovereignty.

## Source-Bound Memory Capsules

Long sessions, phase transitions, handoffs, and resumed agent runs may use a compact source-bound memory capsule, but the capsule is a navigation aid and not a hidden memory runtime. A capsule is valid only when it is recoverable from repo-visible sources or explicitly records that a fact is stale_or_unknown.

The minimum capsule shape is:

- current objective, lifecycle pointers, active phase, stop conditions, and write scope;
- decisions made, docs_decision posture, unresolved blockers, residual risks, and explicit stale_or_unknown markers;
- failed-attempt budget with the verifier or route that failed, the attempt count, and the escalation boundary;
- next safe command or next explicit dry-run route;
- source refs to the state, plan, roadmap item, research, verification, command output, or observation that can be reread.

Provider compaction items, prompt caches, retrieval results, chat summaries, SQLite/projection indexes, MCP output, hook output, dashboard packets, and model memory can help route attention, but they remain non-authority unless the same fact is written through an explicit MLH route into repo-visible files. A memory capsule cannot approve lifecycle movement, closeout, archive, roadmap status, staging, commit, push, provider routing, hidden queues, hidden databases, daemons, or a next-plan opening.

Repeated verifier or route failures must not turn into unbounded retries or quiet success. When the failed-attempt budget is exhausted, the capsule should record deterministic rejection, a human gate, or a bounded handoff with source refs and the next safe command.

Read-only tools may recover lifecycle pointers from a prose-style operating-root `project/project-state.md` when it contains explicit assignment lines such as `plan_status = "active"`, `active_plan = "project/implementation-plan.md"`, `active_phase = "Phase 2"`, and `phase_status = "in_progress"`. This fallback is an analyzer recovery aid. The only mutating bridge from that posture is the bounded `state-frontmatter-repair` class, which snapshots default-path state bytes and prepends deterministic frontmatter before any other repair class can run.

Read-only drift tools may also compare explicit delivered/current capability tokens against future/backlog/remainder wording in operating-memory and research surfaces. That remainder-drift signal is advisory memory hygiene only: it can point to stale backlog wording, but it cannot approve plan edits, closeout, archive, commit, repair, cleanup, or any lifecycle mutation.

## Allowed Product Surfaces

The product source tree may contain:

- `README.md` and `AGENTS.md` orientation surfaces
- product docs under `docs/...`
- product source under `src/mylittleharness/...`
- tests under `tests/...`
- `pyproject.toml` and package metadata
- minimal compatibility fixtures under `.mylittleharness/`, `.agents/`, and `project/` while the CLI/tests need them

## Forbidden Product Memory

The product source tree must not contain:

- active implementation plans
- roadmaps under `project/roadmap.md`
- project-state used as operating memory
- decision/do-not-revisit records under `project/decisions`
- ADR records under `project/adrs`
- research/history/raw intake
- archived plans
- archive-under-study material
- workflow execution state
- runtime debris
- reports, logs, caches, local databases, generated validation artifacts, package archives, build directories, wheels, egg-info, or pycache
- hidden queues, schedulers, dashboards, or control-plane state

## Compatibility Fixtures

Files under `.mylittleharness`, `.agents`, and `project` in the product source checkout are compatibility fixtures unless a later explicit product plan changes their role. Legacy `.codex` manifests are target-root fallback/migration compatibility, not the product fixture path. They support validation and tests; they do not outrank product docs or operating-root workflow authority.

Clean product specs belong under `docs/specs/...`.

## Mutation Authority

An apply flag does not create authority by itself. Mutating CLI behavior may write only repo-visible files inside an explicitly targeted live operating root, and only when the target root, target path, and operation have product-contract authority.

Multi-target apply rails must prepare all replacement bytes before changing live files, then apply bounded file transactions with temporary backups. If a late target replacement fails, completed target writes are rolled back when possible; if rollback cleanup itself fails, the command must report explicitly recoverable residue instead of treating partial mutation as lifecycle approval.

`init --apply` is the primary implemented write gate for attach behavior, with `attach --apply` preserved as the compatibility spelling. Its allowed scope is create-only scaffold setup plus absent `.mylittleharness/project-workflow.toml` and `project/project-state.md` from explicit product templates, followed by attach-time disposable generated projection setup under `.mylittleharness/generated/projection/`. Creating `project/project-state.md` requires an operator-provided project name; the CLI must not infer authoritative project identity from a folder name. Attach apply refuses product-source compatibility fixtures, fallback/archive or generated-output roots, symlink path segments, non-directory scaffold conflicts, non-file template conflicts, template content mismatches, and generated-projection boundary conflicts before writing. SQLite FTS5 or index-build unavailability is degraded generated-output posture, not attach failure.

`repair --apply` is the implemented limited repair write gate. Its allowed scope is snapshot-protected `project/project-state.md` frontmatter prepending for default-path prose operating state, absent eager scaffold directory creation, create-only `AGENTS.md` creation when validation reports a missing required operator contract, create-only `.agents/docmap.yaml` creation when validation reports a missing required docmap, create-only `project/specs/workflow/*.md` restoration when validation reports missing stable spec fixtures, and snapshot-protected `.agents/docmap.yaml` route repair in a live operating root that already has readable workflow-core manifest and strict workflow-core project-state frontmatter authority. Coarse root classification cannot upgrade malformed or partial state into repair authority. It does not create or rewrite manifest, active plans, existing AGENTS contracts, existing stable specs, archives, research files, generated reports, caches, local databases, broad docmap structure beyond the deterministic create template, malformed or partial state frontmatter, or workflow execution surfaces.

`detach --apply` is the implemented marker-only detach write gate. Its allowed scope is create-only `.mylittleharness/detach/disabled.json` in an eligible live operating root with readable default manifest and state authority. The marker is informational evidence only; it does not rewrite authority files, close active plans, delete generated projections, clean snapshots, archive research, mutate Git/config/hooks/CI/package/workstation state, or approve lifecycle decisions.

`writeback --apply` is the implemented closeout and lifecycle write gate for live operating roots. Its allowed scope is the MLH closeout writeback block in `project/project-state.md`, selected lifecycle frontmatter, the marked Current Focus block derived from lifecycle frontmatter, matching active-plan derived copies in explicit Docs Decision, State Transfer, or closeout summary/facts/fields sections, optional same-request roadmap item sync, the default active-plan archive move under `project/archive/plans/`, and operating-memory compaction when the live state file exceeds the default 250-line or 25,000-character threshold. Closeout blocks record plan identity with `plan_id`, `active_plan`, and `archived_plan` when those values are available; partial closeout writeback may carry existing facts only when that recorded identity matches the current active or archived plan, otherwise it refuses unless complete replacement facts are supplied in the same request. A narrow phase-only exception may replace mismatched prior facts when the same request records only `docs_decision = uncertain` and `phase_status = complete`, preserving current plan identity without carrying stale closeout facts. Lifecycle-only phase-pointer writeback does not copy prior closeout facts into a new active plan, and optional roadmap sync uses only same-request closeout facts plus the active or archived plan path. For inactive closeout refresh after archive, `writeback --roadmap-item <id> --archived-plan <project/archive/plans/...>` supplies the archived-plan identity and may update the state closeout block, matching roadmap item, and source-incubation relationship metadata without requiring a readable active plan. `writeback --dry-run|--apply --archived-plan <project/archive/plans/...>` without `--roadmap-item` is an identity-bound archived-plan closeout refresh: the requested path must match `last_archived_plan` or matching project-state/archived-plan closeout identity, complete explicit facts or matching closeout authority must exist, and already synchronized state/archive closeout copies report an explicit no-op. Archived Completed History prose may be reported as compacted done-history evidence, but it does not recreate roadmap item blocks or approve lifecycle movement. `plan --apply` may run the same compaction after its own lifecycle state write, and `writeback --dry-run|--apply --compact-only` exposes only that compaction rail without editing closeout facts, lifecycle frontmatter, active-plan copies, or roadmap fields. Compact-only dry-run reports the full current `project/project-state.md` sha256; compact-only apply requires the matching `--source-hash` when compaction would write and refuses stale or missing hashes before archive/state writes. Compaction may move older state history into `project/archive/reference/project-state-history-YYYY-MM-DD*.md`, but the archive is reference material only and records the source hash; current `project/project-state.md` remains operating memory authority. Read-only context warnings cannot trigger this write path by themselves.

Closeout `next_state` uses actor-neutral decision language. New writeback facts should record `explicit-decision-required` when the next legal step is an explicit authorized decision rather than a command or no-op. `human-decision-required` remains a compatibility alias during migration and is accepted as the same explicit-decision state, but new writes normalize the value to `explicit-decision-required`, and `check` reports legacy closeout facts so they can be migrated before alias removal. This token migration does not grant automation authority, weaken decision gates, satisfy `next_safe` advisory output, approve closeout/archive/roadmap movement, or remove the requirement for a reviewed `legal-dry-run-command:<mylittleharness --root ... --dry-run ...>` when the next state is a concrete route preview.

`task-session --inspect`, `task-session --inspect --fan-in`, and `task-session --dry-run|--apply` are the implemented external-runtime bridge. Inspect writes nothing and cannot be used as launch, provider, lifecycle, route-proposal, or fan-in approval. Fan-in inspect reads the session graph and matching receipts as readiness evidence only; it may report ready or blocked, but it still grants no approval. Receipt apply writes only `project/verification/task-sessions/<session-id>.json`, refuses existing receipt ids, unsafe paths, lifecycle-authority write scopes, and forbidden lifecycle/provider/Git routes, and records all approval fields as false. The receipt route is allowed to bind an optional SDK runtime to MLH preflight evidence, but it does not create a queue, daemon, hidden memory store, worker process, writeback, archive, roadmap transition, or Git mutation.

Checkpoint package receipts are repo-visible evidence for local checkpoint group review, not checkpoint, Git, staging, commit, archive, cleanup, lifecycle, or provider authority. A live operating root may place JSON receipts under `project/verification/checkpoint-packages/*.json` with schema `mylittleharness.checkpoint-package-receipt.v1`. Each receipt names `package_id`, `target_root`, `package_class`, verdict `allowed`, `blocked`, or `unknown`, `docs_decision`, exact `included_paths` and `skipped_paths`, verification refs, source hashes, residual risk, and explicit non-authority wording. Included/skipped paths must be exact root-relative file refs; included paths must exist and be source-hashed, and generated/cache/private/secret/temp/runtime/VCS paths are not valid included checkpoint members. The receipt can explain local package readiness for operator review, but its verdict cannot approve lifecycle movement, archive, roadmap status, staging, commit, push, release, provider routing, cleanup, or target-repo acceptance.

Compaction selection scans the whole state file and preserves current authority plus the latest relevant update. Operators should not manually shrink only the newest state note to satisfy the threshold; the rail is meant to archive older/non-current history.

`plan --apply` is the implemented deterministic active-plan scaffold write gate for live operating roots. Its allowed scope is the default `project/implementation-plan.md` plus selected lifecycle frontmatter in `project/project-state.md`: `operating_mode`, `plan_status`, `active_plan`, `active_phase`, and `phase_status`, along with the marked Current Focus block derived from those values. With `--roadmap-item`, it may also update the requested roadmap item plus roadmap-slice `covered_roadmap_items` to record the active plan relationship. It accepts explicit title/objective/task input, starts generated plans with `docs_decision = "uncertain"`, and records current-phase-only execution metadata: `execution_policy`, `auto_continue`, `stop_conditions`, and `closeout_boundary`. The default is `execution_policy = current-phase-only` and `auto_continue = false`, so the operator executes only the current active phase and stops after repo-visible evidence/state unless a later explicit contract opts into auto-continuation with stop-condition coverage. It refuses product-source fixtures, fallback/archive roots, ambiguous roots, malformed state frontmatter, non-default plan paths, stale inactive plan files, unsafe path targets, active-plan conflicts unless `--update-active` is supplied, invalid requested roadmap links, and destructive recovery instructions before writing. Generated plans are execution scaffolds, not LLM planning, hidden memory, repair approval, closeout approval, archive approval, commit approval, or rollback authority.

When `--roadmap-item` finds advisory slice metadata, plan output may also include bounded synthesis evidence: bundle rationale, split boundary, target-artifact pressure, and phase pressure. That evidence can help an operator see why a plan covers multiple roadmap items, but it does not change the write boundary or promote roadmap output into lifecycle authority.

`plan-cancel --dry-run|--apply` is the scoped accidental-activation cancel rail for live operating roots. Dry-run reports the route writes and a full activation source sha256; apply requires that reviewed `--source-hash` so changes to project-state, the active plan, roadmap, or owned source-incubation note between preview and apply refuse before writes. Simple cancel is only for pending activations before implementation evidence exists. It refuses non-pending active phases and active session/work-claim/agent-run/verification evidence that references the active plan. When eligible, it clears active lifecycle pointers (`plan_status`, `active_plan`, `active_phase`, and `phase_status`), deletes or retains the default active-plan file by explicit flag, restores one roadmap item to `accepted`, clears the roadmap active `related_plan`, and detaches explicit source-incubation notes whose `related_plan` points at `project/implementation-plan.md`. It cannot approve closeout, archive, repair, product-source rollback, staging, commit, or next-plan opening.

`roadmap --apply` is the implemented accepted-work roadmap item write gate for live operating roots. Its allowed scope is one managed `###` item block inside an existing `project/roadmap.md` with a `## Items` section, or one existing compatibility top-level `## RM-...` section for update and relationship-sync operations, plus MLH-owned `Future Execution Slice Queue` compaction when a canonical queue bullet lists only terminal roadmap items, plus live-tail compaction of older detailed `done` canonical item blocks that have archived-plan, verification, and final docs-decision evidence. Compacted done item ids and archived-plan paths are retained under `Archived Completed History`; the newest detailed done tail and any done item without durable evidence stay in `## Items`. It accepts explicit add/update fields, known roadmap statuses, docs decision values, relationship paths, dependencies, supersession links, verification summary, and carry-forward text; dry-run names the target file, action, item id, changed fields, and compacted item ids. `plan --roadmap-item` and `writeback --roadmap-item` reuse that boundary to update explicit item relationships or same-request closeout summaries without broad roadmap rewriting; plan opening can batch the requested item plus roadmap-slice `covered_roadmap_items` and refresh each explicit source-incubation note back to `related_plan = project/implementation-plan.md`, while archive closeout can batch the requested item plus active-plan `covered_roadmap_items` through one transaction. Compatibility RM sections do not expand non-id `slice_members` work-area labels as covered roadmap items. Apply refuses product-source fixtures, fallback/archive roots, ambiguous roots, malformed state frontmatter, missing or malformed roadmap files, duplicate ids, invalid statuses, invalid docs decisions, unsafe path targets, symlink or parent path conflicts, incompatible relationship destinations, missing related item ids, and self-relations before writing. Roadmap output is sequencing evidence only, not repair approval, closeout approval, archive approval, commit approval, rollback authority, or lifecycle approval.

`memory-hygiene --dry-run --scan` is the implemented read-only relationship hygiene and text-input audit report for live operating roots. It can report stale links, missing reciprocal relationships, orphan incubation notes, CLI text-field posture, `## Entry Coverage` completeness for mixed incubation notes, heuristic split suggestions, and coverage-aware archive candidates. For covered candidates, it emits a reviewable proposal with stable candidate ids, archive targets, exact link-repair files, a proposal token, and per-candidate dry-run/apply command shapes; the scan itself still has no matching apply mode. The report writes no files and cannot make semantic guesses authoritative. `writeback --apply` may archive a source incubation note only when closeout/archive was explicitly requested and structural coverage is complete: single-entry notes can be covered by the closeout chain, while mixed notes need terminal Entry Coverage bullets for every dated entry and still fail closed on open questions, unchecked tasks, malformed coverage, unsafe paths, or archive conflicts. Writeback closeout fields are one-line summaries; multiline detail belongs in the active plan or `project/verification/*.md` and is refused on those flags before writes.

Product-source compatibility fixtures are never live operating memory. Apply modes must refuse product-source roots and fixtures with exit code `2`; dry-run modes may continue to report no-op or review-only proposals.

Rollback is not implied by the memory model. Any overwrite, delete, rename, normalization, resync, archive, active-plan mutation, or lifecycle decision operation requires an explicit rollback or snapshot contract before implementation.

Repair snapshots are safety evidence, not operating memory. The repair snapshot boundary is `.mylittleharness/snapshots/repair/` inside an explicitly targeted live operating root, and it may contain only pre-repair file copies plus inspectable metadata for a named repair class. `snapshot --inspect` is a read-only report over that boundary; it can report metadata readability, copied-file/hash/path consistency, target-root confinement, current-target posture, retention, manual rollback instructions, and non-authority wording, but it cannot authorize repair, rollback, cleanup, closeout, archive, commit, lifecycle changes, or future mutations. `repair --dry-run` may report a no-write `state-frontmatter-repair` snapshot plan for default-path prose `project/project-state.md`, create-only plans for a missing required `AGENTS.md`, missing required `.agents/docmap.yaml`, or missing stable workflow spec fixtures, and a no-write `.agents/docmap.yaml` route-repair snapshot plan with deterministic preview paths, metadata fields, copied-file paths, refusal or skip posture, manual rollback posture, and validation commands. Snapshot metadata cannot authorize repair, closeout, archive, commit, plan lifecycle changes, or future mutations.

Snapshot retention is manual-first. MyLittleHarness must not silently delete or rotate repair snapshots; a later cleanup command would need its own dry-run, refusal rules, and product contract. Product-source compatibility fixtures must refuse snapshot creation, and any snapshot directory inside a product source checkout is product debris rather than compatibility data.
