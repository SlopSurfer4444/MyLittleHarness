# Metadata Routing And Evidence Spec

## Purpose

This spec defines how MyLittleHarness uses metadata, routing surfaces, Git-native evidence, and doc-gardening checks without making them stronger than file-visible authority.

Metadata should help agents and tools find the right files, understand lifecycle, and preserve provenance. It must not become the only meaning of a document.

## Authority

Human-readable repo files remain the recovery baseline. Stable product contracts live under `docs/...` in the product root. Active operating-project focus, plan status, and research intake live in the operating root for the repository MyLittleHarness is servicing.

Metadata may route to authority, classify lifecycle, point at source sets, name provenance, and support validation. The surrounding Markdown content must remain meaningful when metadata is missing, stale, or partially invalid.

Evidence has authority only when it points back to inspectable files, commands, observations, accepted docs, or commit metadata. Evidence paperwork alone is not completion.

## Non-Authority

The following are advisory or generated support, not authority:

- YAML frontmatter by itself
- docmaps and routing maps
- persistent generated evidence manifests
- repair snapshot manifests
- backlink or link reports
- stale-pointer reports
- quality-gate summaries
- search results and metadata indexes
- mlhd runtime cache and projection-refresh ledgers
- adapter memory from skills, MCP, browser, IDE, GitHub, CI, hooks, or task runners

Metadata cannot authorize mutation, closeout, repair, or archiving without the matching repo-visible plan and validation evidence. Repair snapshot metadata is included in this rule even when it records copied file hashes and rollback instructions.

## Current Contract

Metadata and routing are allowed when they reduce context load and improve recovery. They must stay lightweight, repo-visible when durable, and subordinate to the files they describe.

Target repository work keeps active memory and new research in that target repository. The product repository receives reusable product specs only. Legacy reference material is not part of default routing.

Routing surfaces should behave like tables of contents or impact maps, not hidden policy engines. Weak semantic matches, stale docmaps, and incomplete metadata may guide reading, but they must not authorize writes.

MLH-owned lifecycle Markdown writers must make route metadata explicit at write time. Research, incubation, evidence, meta-feedback, roadmap, decision, ADR, and verification writes should produce canonical or route-tolerant frontmatter on new files; append/update paths must preserve existing valid frontmatter or prepend a bounded route frontmatter block before writing into a legacy file. Missing frontmatter on routed lifecycle Markdown is a diagnostic and repair/rewrite cue, not approval to infer metadata from prose alone. The bounded repair path is `repair --dry-run|--apply` with repair class `lifecycle-markdown-frontmatter-repair`: it snapshots every affected source file, prepends conservative route frontmatter, dirties existing projection/SQLite cache, and then stops before other repair classes.

Stable-spec posture metadata is stricter than ordinary product-doc metadata. Plan-facing `project/specs/**/*.md` stable specs and product `docs/specs/**/*.md` contract docs that already carry frontmatter must keep `spec_status` and `implementation_posture` explicit; missing only those posture fields is handled by the bounded `repair --dry-run|--apply` class `spec-posture-frontmatter-repair`. That repair snapshots each target, writes only the missing posture keys, dirties existing projection/SQLite cache through the normal repair boundary, and stops before lifecycle movement, docmap normalization, broad all-Markdown rewrites, archive/reference changes, or product docs promotion. Product docs and runbooks outside strict spec routes remain lightweight: audit findings may report missing optional frontmatter, docmap routing gaps, archive/reference exclusions, and suggested route classes, but they do not make every `docs/**/*.md` file a repair target.

`.agents/docmap.yaml` is route-class metadata and an impact-routing aid, not a per-file registry. Adding a new frontmatter-bearing file under an already-known lifecycle route such as `project/research/*.md`, `project/plan-incubation/*.md`, `project/verification/*.md`, `project/decisions/*.md`, `project/adrs/*.md`, or `project/roadmap.md` should not require a docmap rewrite. Update the docmap only when a route class, directory, entrypoint, authority boundary, docs-impact rule, or routing semantic changes.

Discovery packets are a typed `project/research/*.md` evidence profile for pre-plan discovery, not a new authority layer. A packet may identify itself with `schema: "mylittleharness.discovery-packet.v1"` or `source_type: "pre-plan-discovery-packet"` and must use the existing research-gate vocabulary: `quality_status` is `sufficient-for-planning` or `provisional`, and `planning_reliance` is `allowed` or `blocked`. Optional `discovery_status` is descriptive only; `draft`, `contested`, and `blocked` must map to `planning_reliance: "blocked"`. `source_refs`, `source_members`, and `evidence_refs` are source-bound evidence links for roadmap and plan intake, but they cannot browse, choose providers, mutate source, promote roadmap work, open plans, advance phases, archive, stage, commit, push, or approve lifecycle decisions.

Generated relationship graphs, projection artifacts, dashboards, MCP read projections, and SQLite indexes are disposable views over repo-visible files. After an MLH lifecycle write, existing generated projection or SQLite cache must either be marked dirty, rebuilt through the projection rail, or reported as missing/stale/degraded with a next safe rebuild command before first-contact navigation treats it as trusted context. Read-only surfaces may expose that posture, but they do not refresh cache or create lifecycle authority. The mlhd `projection-refresh.json` runtime marker is a disposable receipt for an explicit foreground tick; it may help dashboards explain the last cache attempt, but it is not a durable evidence database, closeout proof, lifecycle authority, or cache-truth override.

The connect/readiness action packet is route metadata, not route authority. It may copy the current lifecycle posture, active plan/phase, docs_decision candidate, docmap status, docs role metadata, recovery-target posture, cache posture, mlhd freshness posture, next legal preview, and state-write requirement into a compact dashboard/check/doctor/MCP/first-contact hook payload so an operator or agent can see the next safe command without reconstructing it from prose. Those fields remain derived from repo-visible state, plans, inventory, disposable runtime/projection markers, and explicit preview/apply rails; the packet cannot record lifecycle truth, approve docs_decision, satisfy closeout evidence, repair docs, archive plans, move roadmap status, make mlhd or SQLite authoritative, or replace source verification.

Command discovery remains a read-only routing aid for legal rails. Broad operator audit terms such as audit, autonomous audit, free swim, product steward, route audit, or workflow friction audit resolve to a read-only loop: start with `check`, inspect with `intelligence`, review hygiene with focused checks or `memory-hygiene --dry-run --scan`, and use `meta-feedback --dry-run` only after a concrete MLH rough edge exists. Multi-agent evidence terms such as stale claim, missing run evidence, worker residue, approval packet, missing handoff ref, or pending approval route to `check --focus agents` and then the explicit `claim`, `evidence --record`, `handoff`, `approval-packet`, or approval-relay inspection rails instead of generic lifecycle or roadmap routes. That routing cannot repair, capture vague ideas, approve lifecycle movement, archive, promote roadmap work, stage, commit, push, delete worker outputs, release claims, deliver approvals, or mutate product source.

## Dependency Gate Policy

MyLittleHarness keeps a stdlib-first dependency gate for product source and package metadata. Runtime and build dependencies stay empty by default; external dependency additions require an approval packet before adoption, and the packet must include package smoke evidence, license/supply-chain review, no telemetry confirmation, and a rollback plan.

Marko, patch-ng, portalocker, and msgspec remain future gated spikes only. They may be researched or named as candidates, but no dependency adoption by default is allowed without a later accepted implementation slice and the same approval packet evidence.

## V2 Governance Metadata

The v2 foundation promotes agent-governance metadata as protocol data, not as a hidden control plane. The first durable contract is to make future reports, manifests, run records, claims, approval packets, and drift proposals source-bound and reviewable.

Structured route and finding metadata should be able to name:

- `route_id`
- `authority_class`
- `mutates`
- `requires_human_gate`
- `gate_class`
- `human_gate_reason`
- `allowed_decisions`
- `suggested_action`
- `advisory_only`

Agent run evidence should be able to name:

- `run_id`
- `parent_run_id`
- `role_id`
- `actor_id`
- `task_id`
- `input_refs`
- `input_hashes`
- `context_tier_refs`
- `output_refs`
- `claimed_paths`
- `commands`
- `verification_refs`
- `status`
- `stop_reason`
- `attempt_budget`
- `attempt_count`
- `repeated_failure_signature`

When known, evidence may also record model/tool provenance such as `provider`, `model_id`, `adapter`, `tool_schema_hash`, and `output_schema_hash`. These fields are evidence about how output was produced. They do not make MLH a provider gateway and do not make provider metadata authoritative.

Agent run records are the first concrete v2 evidence record. The MVP route is `project/verification/agent-runs/*.md`, written only through explicit `evidence --record --dry-run|--apply` in live operating roots. Records must name role, actor, task, assigned scope, runtime, worktree id, input refs, output refs, claimed paths, changed files, commands, verification refs, status, stop reason, attempt budget, docs_decision, residual risk, optional handoff refs, optional claim refs, optional repeated-failure signature, optional provider/model/tool provenance, and source hashes for referenced local files. Missing handoff or claim refs can be recorded as degraded source-bound evidence, but unsafe paths refuse before apply. They are cold evidence inputs, not hot state, and they cannot approve lifecycle transitions, archive, roadmap status, staging, commit, rollback, hidden worker spawning, or next-plan opening. Bare `check` reads them without writing and warns when required metadata is malformed, path-like refs are unsafe, docs_decision is outside the accepted vocabulary, or recorded source hashes have gone stale.

Work claims are the implemented repo-visible claim ledger before broad fan-out. The MVP route is `project/verification/work-claims/*.json`, written only through explicit `claim --dry-run|--apply --action create|extend|release` in live operating roots. Create records name claim id, kind, owner role, owner actor, execution slice, optional worktree/base revision, claimed routes/paths/resources, release condition, and an optional lease from either `--lease-expires-at` or `--ttl` duration shorthand. Extend updates only the lease metadata on an active non-stale claim and refuses stale, released, missing, or malformed targets; release records explicit release evidence. `claim --status --json` exposes the same advisory report shape as text status, including source-bound findings and the boundary that JSON output cannot approve lifecycle decisions.

Claim overlap is scoped to exclusive claim kinds and matching routes, path prefixes, or resources. Claim dry-run reports create, extend, release, stale-lease, malformed, and overlap posture without writing files; apply refuses before writes when those checks fail. The claim ledger coordinates fan-in only: it can warn, block unsafe claim mutations, and feed later evidence or cockpit projections, but it cannot approve closeout, archive, roadmap status, staging, commit, push, rollback, release, hidden worker spawning, or next-plan opening.

The multi-agent coordination record set is intentionally file-first:

- `project/verification/work-claims/*.json` records active, released, stale, or expired scope claims.
- `project/verification/agent-runs/*.md` records durable run evidence and recovery summaries.
- `project/verification/handoffs/*.json` records worker assignment packets and stop conditions.
- `project/verification/session-work/*.json` records session-scoped active work without replacing global lifecycle state.

The accepted contract says claims, runs, handoffs, and session state are coordination evidence. They may route attention, expose conflicts, and feed dashboards or hooks, but they cannot approve lifecycle movement, archive, roadmap status, staging, commit, push, rollback, release, or next-plan opening. Dashboard and daemon views must rebuild from these records plus optional live observations; runtime cache is disposable and must never be the only copy of claim, run, handoff, docs_decision, or closeout evidence.

For agents editing separate Git worktrees, a shared coordination root may be named with `MLH_COORDINATION_ROOT`. That variable is only a routing hint to a live operating root, and it must not be treated as hidden state, lifecycle authority, or an access-control boundary. Claim and agent-run records may include `coordination_root` and `edit_worktree_root` so later operators can see where coordination evidence was written and where source edits occurred. Those fields are evidence only: they cannot create or clean worktrees, write or release claims, write run records, spawn workers, approve lifecycle movement, archive, stage, commit, push, rollback, release, or open the next plan. Product-source fixtures and archive-shaped roots must be refused as shared coordination roots even when they contain workflow-shaped compatibility files.

The multi-agent security threat model is surfaced as read-only `check-multi-agent-security-*` diagnostics. These findings are deliberately evidence and posture, not a new policy engine. They name the file-first authority boundary for claims, runs, handoffs, and session state; the non-authority boundary for hooks, dashboards, `mlhd`, adapters, provider state, logs, and caches; the dispatcher precondition that a handoff, compatible active claim, and planned evidence path must exist before work starts; and the prompt-injection and secret-leakage rule that untrusted repo text, hook args, dashboard inputs, adapter payloads, logs, environment variables, credentials, provider payloads, and source bodies must be reduced to root-relative refs, hashes, and bounded summaries before they become durable evidence. The diagnostics never write coordination records, install hooks, start daemons, launch workers, refresh caches, store secrets, approve lifecycle movement, archive, stage, commit, push, rollback, release, or open a next plan.

Handoff packets are repo-visible worker assignment packets under `project/verification/handoffs/*.json`. The `handoff --dry-run|--apply --action create` route records handoff id, worker, role, execution slice, optional worktree and revision hints, allowed non-lifecycle routes, write scope, read context through evidence refs, stop conditions, required outputs, approval refs, and required work-claim refs. Creation refuses path escapes, missing required evidence or claim refs, duplicate ids, and lifecycle-authority route expansion such as `plan`, `writeback`, `transition`, `roadmap`, `repair`, or other coordinator-owned mutation rails. `handoff --dry-run|--apply --action accept` updates one existing packet to `status=accepted` with an accepting actor and timestamp; acceptance acknowledges the assignment contract but still cannot spawn a worker, approve fan-in, release claims, close lifecycle, archive, stage, commit, push, rollback, or release. `handoff --status`, `check`, and `reconcile` read packets without writing and warn on malformed packet JSON, unsafe write scope, missing or degraded evidence refs, missing or wrong-shape approval packet refs, and missing or wrong-shape work-claim refs.

The dispatcher-agent-launcher slice makes `handoff --status` the first dispatcher readiness surface. It is a read-only gate over existing repo-visible records, not a new queue or hidden runtime. A packet reports launch-ready only when the handoff schema/status is usable, the role id exists in the advisory role manifest, every referenced work claim is a compatible active non-stale `project/verification/work-claims/*.json` record, and every evidence ref is a safe `project/verification/agent-runs/*.md` path. Missing evidence files are allowed as planned run-record targets; existing evidence files must be readable agent-run records. When both handoff write scope and claim paths are present, their path scopes must overlap. Missing packets, missing claims, stale/released/incompatible claims, unsafe refs, malformed existing run records, or refs outside the coordination routes report `dispatcher-launch-refused`. The same status report includes `MLH_COORDINATION_ROOT` diagnostics so external launchers can see the coordination root/edit-worktree boundary. It starts no worker process, writes no run record, stores no provider state, and cannot approve fan-in, lifecycle movement, archive, roadmap status, staging, commit, push, rollback, release, or next-plan opening.

Review tokens and approval packets should bind approval to planned writes and verification, not to an agent's self-report. They should preserve parent operation identity so a risky operation cannot bypass review by being split into many smaller patches.

Approval-packet gate classes are the policy vocabulary for risky operations. Canonical classes are: `lifecycle-authority-mutation`, `write-scope-expansion`, `dependency-package-supply-chain`, `destructive-archive-vcs-rollback`, `external-service-secrets-network`, `fan-in-merge-review-token`, and `repeated-verifier-failure-or-uncertain-evidence`. A packet may use a narrower class when a later spec defines it, but broad risky work should map back to one of these classes before an apply rail treats the packet as review evidence.

Approval packets must be replayable without chat memory. The minimal packet shape is: packet id, requester or actor/session, timestamp, lifecycle subject, requested decision, gate class, risk class, validity or expiry, target artifacts, planned writes and boundaries, allowed command/network/auth scopes, expected blast radius, source refs, dry-run refs, verifier refs, diff or patch hash or base/head identity when available, review-token hash when available, docs and lifecycle impact, stop or reopen conditions, human decision, approver identity, decision timestamp, fallback or rejection reason, and residual risk. Packet status is evidence about a reviewed request, not proof that the work is correct or that lifecycle, archive, roadmap, Git, release, relay delivery, or dependency adoption happened.

Route receipts are protocol/report data only. Role manifests may name `work_claim_required`, `work_claim_contract`, `route_receipt_contract`, `fan_in_authority`, and `runtime_boundary` so coordinators can inspect fan-in requirements before any worker packet is trusted. The coordinator retains lifecycle authority; worker packets are evidence only, and work claims, review tokens, and route receipts remain evidence, not apply authority. Read-only protocol and coordination reports such as `manifest --inspect`, `claim --status`, `reconcile`, and `review-token` should render explicit no-write Work Result capsules so report completion is not mistaken for claim release, fan-in approval, lifecycle approval, archive approval, staging, commit, push, or release authority. Review tokens bind the active-plan route only when project-state explicitly names `active_plan` or records `plan_status = active`; an inactive default `project/implementation-plan.md` route is not fan-in input by itself. Review-token ref inputs must resolve to readable repo-visible files before token trust; missing, unreadable, or non-regular claim/evidence/human-gate refs refuse before a token is emitted, while intentionally external or summarized evidence can be bound through explicit reviewed hash inputs. Refused read-only reports should say to rerun the read-only command after fixing inputs, not to rerun a dry-run or apply. Coordination mutation previews must keep dry-run refusal summaries distinct from apply refusal summaries, so a refused `claim --dry-run`, `handoff --dry-run`, or `approval-packet --dry-run` cannot be read as evidence that an apply was attempted. Approval packets are append-only evidence; existing packet ids refuse with a supersession route that asks for a new approval id and includes the prior packet as an input ref. Approval packets with `status=approved` are still evidence packets: Work Result wording must not let the status stand in for the human decision, mutate an existing packet's status, or approve lifecycle/archive/Git/release decisions. Command discovery must keep worker handoff packet wording on `handoff --dry-run` and work-claim creation wording on `claim --dry-run --action create` rather than routing those queries to lifecycle phase-handoff or plan-opening rails. The runtime boundary must continue to reject worker lifecycle writes, hidden daemons, model calls, provider gateways, staging, commits, pushes, archive approval, roadmap status changes, and next-plan opening.

Reconcile output is proposal metadata only. It may classify a spec/code/evidence relation as `unassessed`, `partially_verified`, `synced`, `stale`, or `drift_detected`, and it may propose amendment, supersession, verification-needed, or drift-record artifacts. The implemented `reconcile` command and `check --focus agents` report route, source, and evidence fingerprints from repo-visible paths and agent-run source hashes; they also surface malformed work-claim records, stale work claims, overlapping active work claims, missing run evidence, stale approval packets, malformed/degraded handoff packet refs, and unclaimed worker-space residue. `claim --status` and agent-focused checks must warn when repo-visible work-claim evidence is missing required identity/owner/scope fields, uses unsupported status/kind values, stores scope lists in a non-list shape, carries unsafe claimed paths, has an unparsable lease timestamp, or leaves two active exclusive claims overlapping on routes, paths, or resources. Handoff packet diagnostics warn on malformed existing packet JSON plus missing, unreadable, non-regular, or wrong-route evidence, approval-packet, and work-claim refs; those warnings are review cues only and cannot approve worker fan-in, cleanup, release, or lifecycle movement. The manifest default active-plan route is assessed only when `project-state` records `plan_status = active` or explicitly names `active_plan`; when `plan_status = none` and `active_plan` is empty, reconcile reports the implementation-plan route as lazy or inactive instead of warning that it is missing. They must not silently edit accepted specs to match implementation, release claims, delete worktrees, stage, commit, archive, or approve lifecycle movement. When those statuses appear in route metadata, `check` may report the state and a next safe command, but every amendment remains human-gated and no status report is implementation approval by itself.

Spec lifecycle posture uses two separate fields when a stable spec or `docs/specs/**` product spec is plan-facing or explicitly opts into posture metadata: `spec_status` describes the spec document lifecycle (`draft`, `accepted`, `superseded`, or `archived`), while `implementation_posture` describes current implementation evidence (`not-applicable`, `target-only`, `in-progress`, `partially-verified`, `synced`, `drift-detected`, `deprecated-compat`, or `retired`). `docs_decision` stays a closeout-local decision and must not be reused as spec lifecycle. Read-only validation may emit `spec-posture-missing`, `spec-synced-without-verification`, `spec-target-only-has-implementation-evidence`, `spec-drift-detected-without-carry-forward`, or `spec-superseded-without-target`; those findings are reconcile cues only. An accepted `target-only` spec is preserved as target authority and must not be deleted merely because code has not caught up. `synced` requires source-bound verification or closeout evidence, drift requires carry-forward/amendment/replan evidence, and superseded/deprecated/retired specs require a named replacement, deprecation, retirement, or archive path. All amendments remain human-gated.

## Intake Routing

Incoming information may be classified before it becomes incubation clutter. Hidden `intake --dry-run` accepts explicit text or UTF-8 `--text-file` input and reports one route advisor classification for future ideas, research imports, decisions, ADRs, verification evidence, docs impact, archive references, or ambiguous input. The report is advisory only.

`intake --apply` is intentionally narrower than the classifier. It requires a live operating root, unambiguous classification, and an explicit new root-relative Markdown `--target` whose route matches the classifier. It refuses product fixtures, fallback/archive roots, path escapes, symlinks, existing targets, ambiguous text, and mismatched destinations before writing. Apply writes only that one target file and cannot approve roadmap promotion, repair, closeout, archive, commit, rollback, or lifecycle decisions.

## Research Artifact Rails

Research routes are evidence containers, not researcher minds. `research-import` may place external research or agent-authored research notes in `project/research/` with provenance. `research-distill`, when present, is a distillate intake and validation rail: it may scaffold a reviewable container, preserve source routes, check claim-to-evidence coverage, require contradiction/uncertainty/confidence fields, mark shallow material as provisional or insufficient, and block downstream planning reliance until a human or agent supplies an adequate distillate. `research-compare` may write comparison artifacts with `status: "compared"`; that status is valid research route metadata for a non-authority comparison artifact, not a lifecycle approval or accepted-research decision. Generated research distillates and comparison artifacts omit empty optional relationship fields instead of writing blank scalar lifecycle hints such as `superseded_by: ""`; absence means no reviewed relationship, not an implicit lifecycle decision. Research import reports include read-only lifecycle provenance posture so active lifecycle or writeback facts without agent-run evidence are visible during intake without blocking import or approving closeout.

MLH must not turn source text into substantive conclusions, resolve contradictions, assign truth, write qualitative synthesis, or close roadmap/plan gates merely because a Markdown distillate is syntactically valid. The active agent or human researcher owns prompt design, source acquisition, interpretation, and synthesis; MLH owns routing, provenance, linting, warnings, and lifecycle boundaries.

## Relationship Hygiene

Lifecycle relationship hygiene uses a minimal path/id vocabulary rather than a separate authoritative index. Accepted fields include provenance links such as `source_incubation`, `related_incubation`, `source_research`, and typed `source_members`, roadmap links such as `related_roadmap` and `related_roadmap_item`, plan links such as `related_plan`, `archived_plan`, and `implemented_by`, supporting links such as `related_spec`, `related_decision`, `related_adr`, and `related_verification`, closeout facts such as `verification_summary`, `docs_decision`, and `carry_forward`, archive and promotion links such as `archived_to` and `promoted_to`, and lifecycle relation fields such as `supersedes`, `superseded_by`, `merged_into`, `merged_from`, `split_from`, `split_to`, `rejected_by`, and `rejection_reason`. Detached incubation and meta-feedback notes may omit roadmap and plan relationship fields until a reviewed roadmap, plan, writeback, or reconciliation rail owns that relationship; docmaps should model those fields as optional relationship metadata rather than universal required frontmatter.

Read-only relationship hygiene can be surfaced through `check --focus hygiene`, `check --deep`, or `memory-hygiene --dry-run --scan`. It may report stale relationship paths, missing accepted/active roadmap `source_incubation` evidence before plan opening, missing reciprocal roadmap/source-incubation links, orphan incubation notes, done roadmap items without archive or verification links except explicit terminal historical relationship stubs owned by archive-context evidence, implemented notes still in active incubation, coverage-aware archive candidates, Entry Coverage blockers, incubation cleanup advisor classifications for keep-active notes, follow-up extraction, archive candidates, exact-link repair candidates, and ambiguous notes, Deep Research rubric recovery hints when rubric cues exist but no durable research/reference rubric artifact is recoverable, CLI text-input audit posture, heuristic split suggestions, and reviewable batch-proposal details for covered cleanup candidates. Batch-proposal details include candidate ids, source hashes, archive targets, exact link-repair file hashes, a proposal token, and a copy-ready `memory-hygiene --apply --scan --proposal-token <mhb-token>` command, but the dry-run scan remains a diagnostic report until that explicit apply command succeeds.

Canonical meta-feedback cluster notes under `project/plan-incubation/` are intentionally roadmap-detached operating memory until explicit promotion. Relationship hygiene keeps those `[MLH-Fix-Candidate]` cluster notes active without reporting them as orphan or ambiguous solely because no roadmap/plan/archive relationship exists yet, and it does not require Entry Coverage or split cleanup solely because the active cluster has multiple dated observations.

Done roadmap items whose `archived_plan` points at an explicitly reconstructed historical archive card may keep `docs_decision: uncertain` as an informational boundary instead of a fresh missing-docs warning. That posture preserves provenance for recovered dependency evidence; it does not infer a final docs decision, approve closeout, or remove the need to re-review docs when a current lifecycle decision depends on that historical slice.

`incubation-reconcile --dry-run|--apply` is the bounded reconciliation rail for live `project/plan-incubation/*.md` notes. Dry-run classifies selected notes as `active-roadmap-source`, `archived-covered`, `promoted-compacted`, `orphan-needs-triage`, `duplicate-or-superseded`, or `still-live-followup` from repo-visible roadmap/source/archive metadata. Apply writes only reconciliation frontmatter fields (`lifecycle_status`, `resolution`, `resolved_by`, `superseded_by`, and `last_reconciled`) to the selected notes. It does not delete notes, change roadmap status, repair links, archive anything, stage, commit, or approve lifecycle movement.

Meta-feedback capture is operating-memory evidence, not lifecycle authority. The `meta-feedback` rail writes canonical incubation notes only in the central MyLittleHarness-dev live operating root, using `--to-root <central-root>`, `MYLITTLEHARNESS_META_FEEDBACK_ROOT`, or `--root` when it already points at that central root; noncentral serviced or product roots remain observed-root provenance and are refused as canonical MLH product-debt destinations. Reports name whether the destination came from `--to-root`, `MYLITTLEHARNESS_META_FEEDBACK_ROOT`, or `--root`; when the environment variable is used, the report says so explicitly, and when `--to-root` overrides an ambient environment destination, the report names that override. Canonical notes carry write-route provenance fields for capture mode, requested root, destination root/source, `--to-root`, ambient env destination, note source, and the non-enforced dry-run/apply pairing boundary, plus managed cluster metadata such as `canonical_id`, `friction_signature`, `occurrence_count`, `first_seen`, `last_seen`, `observed_roots`, `affected_routes`, `duplicate_topics`, `recurrence_score`, representative examples, and `latest_observation_hash`; exact-signature duplicates can append automatically and `--dedupe-to <canonical-id>` supports explicit near-duplicate appends. When `--roadmap-item <canonical-id>` is supplied, that id is the canonical note slug and cluster target even if `--topic` is more human-readable; the topic is retained as duplicate-topic/representative evidence rather than creating a side note that can drift from the cluster metadata target. `--correction-of latest|<observation-hash>` requires an existing canonical note, appends an explicit correction marker, reports before/after observation hashes, leaves `occurrence_count` unchanged, and increments correction metadata instead of treating metadata repair as stronger product signal. When explicit flags are absent, structured note fields such as `signal_type`, `severity`, `affected_routes`, and `expected_owner_command` feed the managed note and cluster metadata so those fields do not diverge from the pasted observation; inline command code spans in structured values are preserved in the note body and normalized only where the managed metadata scalar format cannot safely contain raw backticks. `meta-feedback` does not add or update matching accepted roadmap items; mature clusters must be promoted through explicit `roadmap --dry-run|--apply` review. Roadmap-detached reporting is part of the command boundary and cannot approve repair, closeout, archive, lifecycle movement, release cleanup, staging, commit, push, roadmap promotion, or next-plan opening. Apply reports include post-apply lifecycle posture evidence: if `plan_status`, `active_plan`, phase fields, or active-plan file posture move during the command, the report warns with before/after values and routes the operator back to `check` and repo-visible route inspection instead of silently treating the movement as approved.

Same-transaction writeback is allowed only for facts structurally known from the requested operation. `incubate --apply` may record the current active plan relationship, and the active plan's roadmap item when present, only in newly created incubation notes whose topic or slug matches the current plan identity, primary roadmap item, execution slice, or covered roadmap items; unrelated recovery or future-idea notes stay unclaimed. ## Validation Expectations

A valid implementation should prove that:

- documents remain understandable without metadata
- metadata routes to existing files
- stale metadata is reported rather than silently trusted
- spec_status and implementation_posture stay separate, target-only specs are preserved, and supersession/deletion remain explicit human-gated lifecycle decisions
- evidence points to inspectable artifacts or explicit skip rationale
- terminal evidence reports remain useful without storing generated truth
- report-only evidence cue identity remains source-bound and deterministic without becoming a manifest
- durable proof/evidence records under `project/verification/*.md` are surfaced as optional report inputs without satisfying closeout fields by themselves
- manifest or prospective closeout wording remains context rather than concrete closeout completion evidence
- closeout assembly reports preserve Git/VCS facts and any suggested trailers as advisory evidence rather than required authority
- report-only quality gates improve closeout readiness without enforcement state
- generated reports can be deleted and rebuilt when a later generated-report feature exists
- routing never authorizes mutation by itself
- repair snapshot metadata and `snapshot --inspect` remain evidence for copied bytes, not authority for repair, rollback, cleanup, or closeout
- writeback apply synchronizes closeout/state authority, the marked Current Focus block, and derived active-plan copies in the same mutation, while read-only commands keep reporting stale derived copies without writing
- archive-active-plan writeback dry-run/apply closes the default completed-plan lifecycle, and when matching project-state closeout authority is already complete it carries those facts into the archived-plan copy and retargets closeout identity in the same transaction, without turning repair, check, evidence, or closeout reports into archive approval
- post-writeback, post-plan, and compact-only auto-compaction keep oversized live operating memory compact without giving read-only context warnings permission to write files
- roadmap dry-run/apply can add or update one explicit managed item block, compact eligible completed queue/live-tail history into archived references, and preserve roadmap authority boundaries
- plan/writeback roadmap sync can update an explicit roadmap relationship, and writeback archive closeout can batch active-plan covered roadmap items, without making roadmap output lifecycle approval
- plan --roadmap-item can derive active-plan slice frontmatter from repo-visible roadmap fields without putting slice membership into project-state
- live-root status/check summarize completed active phases as explicit closeout/archive/manual-commit next action rather than leaving agents to reconstruct that state from prose history
- legacy reference material is opened only for a named blocker
- multi-agent security diagnostics expose unsafe-default, prompt-injection, path/root, adapter, hook, dashboard, daemon, dispatcher, and secret-leakage posture without creating hidden runtime state

Validation should prefer report-first checks. Mutating doc repair requires a separate explicit plan.

## Explicit Non-Goals

- No metadata schema implementation in this phase.
- No evidence database, hidden registry, default persistent evidence manifest, or generated truth.
- No Git staging, commits, amendments, pushes, hook installation, or Git config mutation by evidence helpers.
- No required docmap for a first operating-root phase.
- No auto-gardening or silent doc repair.
- No metadata-only authority.
- No quality-gate ceremony for every small ad hoc task.
- No import of old research or archives as routing defaults.
