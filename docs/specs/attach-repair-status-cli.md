# Attach Repair Status CLI Spec

## Purpose

This spec defines the product boundary for MyLittleHarness CLI surfaces that attach, check, repair, detach, and expose advanced diagnostics for repository contracts.

The CLI contract must be explicit before product code implements commands that inspect or mutate operating memory, fixtures, or reusable product docs.

## Authority

Repo-visible files remain authoritative. A CLI may discover, summarize, validate, or propose changes, but accepted workflow truth remains in the target repository operating root and stable product contracts.

For the portable clean-room posture:

- the operating project root is the repository MyLittleHarness is servicing and owns active operating memory and plans for that repository
- the product source checkout owns reusable product source and docs
- legacy reference material is opened only for a named blocker
- MyLittleHarness serves the explicit target repository directly

The CLI must preserve the distinction between live operating memory and product-root compatibility fixtures.

## Non-Authority

CLI output is not authority by itself. Diagnostics, generated reports, dry-run plans, stdout, exit codes, caches, and SQLite rows must not hold the only copy of accepted decisions, active focus, plan status, or stable workflow rules.

A successful status or validate run does not approve lifecycle decision, close a plan, archive history, or authorize repair. A repair proposal does not become accepted until a later mutating path writes repo-visible files under an explicit plan.

## Target Visible CLI

The target product CLI is a small repo utility:

- `init`: attach MyLittleHarness to an explicit target repository. Current implementation routes `init --dry-run` and `init --apply --project <name>` through the same bounded attach behavior as the compatibility `attach` command.
- `check`: report orientation, validation, compact check-level drift, and live-root lifecycle routes without writing. Current implementation composes `status`, `validate`, and a small docmap/root-pointer drift section; deeper diagnostics remain explicit advanced commands.
- `repair`: preview or apply one bounded repair class at a time. Current `repair --dry-run` and `repair --apply` remain the product safety model.
- `detach`: preview harness disconnect posture and apply a marker-only disabled posture with no surprise deletion. Current `detach --dry-run` is read-only and preservation-first; `detach --apply` creates only the disabled marker in eligible live operating roots.

First-contact docs and top-level help show this small shape first. The first-run operator path starts from source-checkout usage with `PYTHONPATH=src`, optionally verifies package metadata and the console script with `bootstrap --package-smoke`, then points `--root` at the target repository for `init --dry-run`, `check`, `repair --dry-run`, and `detach --dry-run`. Advanced diagnostics may remain available through compatibility commands, hidden/deemphasized help, or an explicit advanced namespace, but they should not define the product front door.

The no-skill start pass is part of the CLI contract. A file-reading, shell-capable agent reads `AGENTS.md`, `.codex/project-workflow.toml`, `project/project-state.md`, and `project/implementation-plan.md` only when `plan_status = "active"` or the operator asks about plan/phase/closeout, then runs `check` before mutation. `.agents/docmap.yaml`, `audit-links`, `check`, and relevant specs provide portable docs-decision inputs; no `docs-impact`, `guide`, or `orient` command is required for v1.

The optional agent-accelerator start pass is helper-owned and explicit. `adapter --client-config --target mcp-read-projection` previews the rootless Codex MCP config payload, mount posture, first-pass commands, and no-secret boundary without writing. `adapter --install-client-config --target mcp-read-projection --dry-run|--apply` is the only implemented user-global client config mutation: apply may touch only the reviewed Codex config target, writes an idempotent managed MCP server table, backs up existing config before append or replacement, replaces the legacy root-bound MLH server table only when it exactly matches the old managed command shape, refuses unmanaged conflicts or unreadable/invalid config, and must not print existing config values. `hooks adapter --client codex --dry-run|--apply --scope project` is the implemented project-local native hook activation rail: apply may write only `.codex/hooks.json` plus `.codex/hooks/mylittleharness_session_start.py` to register a Codex `SessionStart` command hook. The helper derives context from the full `hooks --run session-start --json` payload but writes only Codex's minimal SessionStart command-output envelope to stdout. `dashboard --inspect --json`, bare `check`, and `hooks --run session-start --json` expose whether the dashboard packet is available, whether MCP is mounted/missing/legacy-root-bound/conflicting, whether projection cache can be self-healed with `projection --warm-cache --target all`, and that direct `rg` or bounded file verification remains required after any accelerator path.

`sync` is not a primary command. Disposable projection refresh, search, semantic readiness, adapter projection, and report assembly stay advanced unless a later implementation proves a visible refresh command reduces operator ceremony.

Compatibility is a transition policy:

- Existing commands and exit semantics must remain intact during the first CLI simplification slice.
- Old commands may become aliases, advanced commands, or compatibility surfaces after primary `init` and `check` exist.
- Deprecation starts in docs/help. Normal command stdout should not gain churny warnings unless a later versioned deprecation slice explicitly accepts that behavior.
- Command removal requires a later scoped deprecation/removal slice with tests and migration notes.

## Current Contract

The implemented 1.0.0 product contract is:

- Text reports and JSON reports carry a `Work Result`/`work_result` capsule that includes outcome, practical change summary, an adaptive ceremony cost/guarantee line, and the next safe command or route-discovery step. These report fields are advisory ergonomics: they may point operators at dry-run/apply rails or `suggest --intent`, but they cannot approve repair, lifecycle movement, archive, staging, commit, push, rollback, or future mutations. Read-only inspect/status/fan-in reports that write no files must use the explicit read-only change line and advisory ceremony guarantee rather than the weaker generic "no file-level changes reported" wording; read-only reports with warnings must not say authority is unchanged "until an explicit apply succeeds" unless the report is actually a preview for a matching apply. No-op previews, such as `repair --dry-run` with no repairable diagnostics or `writeback --dry-run --compact-only` below the compaction threshold, must say no apply target exists instead of advertising an empty apply follow-up. Error summaries and ceremony lines must stay mode-specific: preview refusals, read-only refusals, and apply refusals should not borrow each other's wording.
- Advanced reports share a rails-not-cognition boundary: MLH reports route, validation, scaffold, provenance, comparison, retrieval, and evidence candidates only. Source files and explicit operator or human-reviewed lifecycle decisions remain authority; wording such as distill, compare, derive, classify, recommend, rank, and suggest must be framed as candidate/provisional/report-only output, not as product advice, truth selection, priority, or lifecycle approval.
- `init --dry-run` and `init --apply --project <name>` are the primary spellings for bounded scaffold attachment. They preserve the existing attach dry-run/apply output shape, mutation boundary, and exit behavior. Valid already-attached live roots report `attach-already-attached` as a clean no-op instead of first-run template conflicts, and already-attached apply skips generated projection writes. First-time successful attach still builds disposable generated projection artifacts and the SQLite index.
- `check` is read-only. Bare `check` is sectioned as `Status`, `Validation`, `Agent Run Evidence`, `Work Claims`, `Handoff Packets`, `Projection Cache`, `Drift`, and `Boundary`. It composes `status`, `validate`, agent run record diagnostics, work-claim status, handoff packet ref diagnostics, projection-cache posture, and check-level drift findings for candidate docmap gaps, stale configured-root role wording, primary instruction-surface size warnings, current project-state prose that appears to disagree with lifecycle frontmatter, and explicit delivered-vs-remainder capability token contradictions in operating-memory or research surfaces, writes no files, returns `1` when included findings have error severity, and otherwise returns `0` even when warning findings are present. For live operating roots, `Status` includes the compact lifecycle route table, including optional `project/roadmap.md` sequencing when planning context needs it; product-source fixtures do not emit live route-table rows. Validation also reports `roadmap-source-incubation-missing` when accepted or active roadmap items point at missing `source_incubation` evidence before operators rely on roadmap-derived plan input, and `roadmap-related-spec-missing` when accepted or active roadmap `related_specs` point at absent stable-spec files that would otherwise leak into generated plan docs/read/write scope; those findings suggest source-note recovery, spec retarget/removal, or provisional `docs_decision='uncertain'` and remain read-only. When Deep Research rubric cues are present but no durable comparison-rubric artifact is recoverable from `project/research/harness-deep-research-comparison-rubric.md` or `project/archive/reference/research/**`, validation reports `deep-research-rubric-recovery-route-gap` and points at explicit `research-import --dry-run` plus `research-distill --dry-run` before any import apply. The primary instruction-surface size warning is limited to start-pass surfaces such as `AGENTS.md`, `README.md`, `.agents/docmap.yaml`, `.codex/project-workflow.toml`, `project/project-state.md`, and the active plan; product docs and stable specs remain covered by advanced `context-budget` detail instead of check-level drift, and their large measurements are informational unless the same surface is also a primary instruction surface. Oversized live `project/project-state.md` warnings may point operators at whole-state `writeback --dry-run --compact-only`, and `state-lifecycle-prose-drift` asks operators to rewrite stale current prose as history or refresh managed focus through writeback/plan apply, but `check` itself remains no-write.
- `check --deep` is read-only and renders `Status`, `Validation`, `Drift`, `Links`, `Context`, `Hygiene`, `Grain`, and `Boundary`. It consolidates the common read-only compatibility diagnostics behind `check` while preserving the old command spellings.
- `check --focus validation|links|context|hygiene|grain` is read-only and renders exactly one focused diagnostic section plus `Boundary`. The focus values route to `validate`, `audit-links`, `context-budget`, `doctor`, and report-only grain diagnostics respectively, with old wrappers preserved for compatibility.
- `manifest --inspect` is read-only and reports the structured route and role manifest for external agents and orchestrators. Text output has separate `Route Manifest` and `Role Profiles` sections. `manifest --inspect --json` emits `route_manifest` plus `role_manifest`; role entries include permissions, forbidden actions, required outputs, context/output packet requirements, gate classes, and human gates. Role profiles are advisory packet law only: they grant no direct lifecycle apply authority, spawn no workers, and cannot approve repair, closeout, archive, roadmap movement, staging, commit, push, release, or lifecycle transitions.
- Coordination read-only reports such as `claim --status`, `reconcile`, and `review-token` are terminal advisory evidence. Their Work Result capsules should say that no repository files were changed, keep repo-visible files authoritative, and avoid treating "completed" output as claim release, fan-in approval, lifecycle approval, archive approval, staging, commit, push, or release authority. Review-token input digests include the active-plan route only when project-state explicitly names `active_plan` or records `plan_status = active`, so inactive default implementation-plan files do not create accidental fan-in token drift. Review-token ref inputs must resolve to readable repo-visible files before a token is emitted; missing, unreadable, or non-regular refs refuse before token trust, while explicit `--claim-hash`, `--evidence-hash`, and `--human-gate-hash` remain available for reviewed non-file evidence. Work-claim status and agent-focused checks also warn on malformed records and overlapping active exclusive claims so stale, conflicting, or hand-edited coordination evidence cannot look clean merely because a JSON file exists. Existing handoff packets are checked read-only in bare `check`, `check --focus agents`, and `reconcile`; degraded evidence, approval-packet, and work-claim refs are warnings, not worker fan-in approval or cleanup authorization.
- Coordination mutation previews such as `claim --dry-run`, `handoff --dry-run`, and `approval-packet --dry-run` must keep refusal summaries mode-accurate: a refused dry-run reports a refused preview, not a refused apply, while still naming the matching apply as a later explicit route after review. Approval packets are append-only evidence: when an existing `--approval-id` is supplied, the report refuses that route and suggests creating a new approval packet id with the prior packet as an `--input-ref` rather than hand-editing status. When an approval packet is previewed or written with `status=approved`, the Work Result must say that the route records repo-visible human-gate evidence only and cannot grant approval, transition an existing packet, or approve lifecycle/archive/Git/release decisions.
- Coordination command discovery keeps worker packet language off lifecycle handoff rails. `suggest --intent` routes work-claim creation wording to `claim --dry-run --action create` and worker handoff packet wording to `handoff --dry-run`, while `phase-closeout-handoff` remains reserved for same-plan lifecycle phase handoff and archive-close sequencing.
- `suggest --intent "<operator-action>"` and `suggest --list` are read-only command-discovery reports over the deterministic command intent registry. Text output renders `Command Suggestions` plus `Boundary`; `--json` emits `command_suggestions` with intent ids, aliases, first safe commands, follow-up commands, root posture, and boundary language, plus a `rails_not_cognition` boundary flag. Command templates quote free-text placeholders such as query, topic, note, task, condition, and closeout text values, while root-relative route/path/id placeholders stay unquoted. Agent-navigation reflex wording such as route discovery, Chutye/reflex, repo-intelligence defaults, MCP/rg navigation, event-driven self-check, or chat-output-policy recovery routes first to `intelligence --query "<topic-or-route-question>"`, then route-focus intelligence, MCP client-config inspection, and `suggest --intent "<operator-action>"` as follow-ups. Projection cache stale/dirty/rebuild wording routes to `projection --inspect --target all`, then explicit generated-cache `projection --rebuild --target all` or an intelligence search refresh, without implying source mutation. Missing roadmap `source_incubation` or relationship-writeback refusal wording routes to the recovery scan first, then explicit incubation or roadmap retarget dry-runs. Missing `docs/specs/...`, product-doc, backlink, fan-in, or intelligence-warning wording routes to docs-route recovery triage first, then link audit, focused link checks, or meta-feedback capture; it must not be reclassified as Deep Research or roadmap source-incubation recovery solely because the missing docs path contains words such as research or prompt. Missing `project/research/...`, `source_research`, distillate, or research provenance wording routes to research-route recovery triage first, then link audit, focused link checks, memory-hygiene scan, explicit `research-import --dry-run`, or meta-feedback capture; it must not be reclassified as docs, Deep Research prompt/rubric, or roadmap source-incubation recovery solely because the query contains generic recover/missing/research terms. Deep Research prompt, simple English or Russian prompt requests, `needs_deep_research`, `requires_reflection`, and needs-human-review wording route to a pause for manual external prompt drafting outside MyLittleHarness, then `research-import`, `research-distill`, or an explicit roadmap update after the external research exists. Deep Research rubric recovery wording routes to `check`, `memory-hygiene --dry-run --scan`, and explicit `research-import`/`research-distill` previews for `project/research/harness-deep-research-comparison-rubric.md`. command intent suggestions are advisory command-route candidates only: they do not provide product advice, execute suggested commands, write files, approve repair, approve lifecycle movement, archive, stage, commit, push, or mutate workstation state.
- `research-import`, `research-distill`, and `research-compare` are hidden research write rails for non-authority provenance, source-candidate distillates, and comparison matrices. `research-distill` extracts source-candidate, gap, provenance, and route-reference signals from one research artifact; labels such as "accepted candidates" mean extracted from source language, not accepted or prioritized by MLH. `research-compare --dry-run` previews compared sources, source hashes, conflicts, unresolved gaps, route proposals, and route writes without writing files. With explicit `--archive-sources`, apply writes the comparison artifact first, writes archive copies of each compared source with `status`, `promoted_to`, `archived_to`, and `updated` metadata, then deletes the live source files in the same transaction; `--repair-links` rewrites only exact root-relative source-path references to those archive paths and is refused without `--archive-sources`. The comparison artifact preserves unresolved gaps, source hashes, original-source provenance, and archive references, but it cannot choose truth, rank importance, or promote research to specs, plans, roadmap status, closeout facts, staging, commit, or next-plan opening.
- `evidence --record --dry-run|--apply` is an explicit live-root evidence write rail for source-bound agent run records. It requires `--record-id`, role, actor, task, status, stop reason, attempt budget, input refs, output refs, claimed paths, and commands; it may include repeated failure signature plus provider/model/tool provenance. Apply writes only one Markdown record under `project/verification/agent-runs/*.md`, computes source hashes for referenced root-relative files, refuses unsafe paths and existing record ids, creates no hidden runtime, queue, database, cache, adapter state, or provider gateway, and does not approve closeout, archive, roadmap status, staging, commit, rollback, or lifecycle transitions. Bare `check` reports malformed or stale agent run records read-only.
- Link diagnostics classify product-source target artifact references in live operating roots without requiring those product files to exist in the operating root. Paths under product-source context, including `target_artifacts` metadata for `src/`, `tests/`, `docs/`, `build_backend/`, `README.md`, `AGENTS.md`, or `pyproject.toml`, report `product-target-artifact` rather than generic `missing-link`; unrelated missing local links keep their warning posture. The shared target-artifact ownership resolver used by `plan`, `roadmap`, and `check` classifies roadmap or active-plan `target_artifacts` as `product-source-artifact`, `operating-memory-route`, `product-compat-fixture`, `generated-cache`, `archive-evidence`, or `unknown`, reports the intended write root such as `product_source_root=<path>` or `operating-root`, and remains read-only guidance with no automatic mirror or product mutation.
- Link diagnostics also classify missing `docs/specs/research-prompt-packets.md` references in imported `project/research/**` research-prompt packet/rubric context as `historical-link-context`. That retired packet surface should not keep warning through `intelligence --focus warnings` or fan-in solely because archived research evidence still names it; restoring or recreating it remains a separate reviewed docs route decision.
- Grain diagnostics inspect active-plan frontmatter, covered roadmap items, target artifacts, phase count, write-scope and verification specificity, unsafe auto-continue posture, roadmap done/evidence hygiene, targetless non-closed roadmap items, and recent archived-plan calibration samples. These findings are sizing and hygiene signals only: they do not write files, infer roadmap status, approve repair, closeout, archive, commit, rollback, lifecycle decisions, or promote numeric thresholds into product gates.
- `roadmap --dry-run --action add --item-id <id> --title "<title>" --status <status> --order <int>` and `roadmap --dry-run --action update --item-id <id> [fields]` report the target roadmap file, action, item id, changed fields, reciprocal source-incubation relationship writes, relationship frontmatter block recovery, refusal posture, and write boundary without writing files. `roadmap --apply` writes only `project/roadmap.md` plus the explicit `source_incubation` note frontmatter in a live operating root by appending one managed canonical item block, replacing one existing canonical item block, or updating one existing compatibility top-level `## RM-...` section. Canonical roadmaps refresh the MLH-owned `Future Execution Slice Queue` from accepted/proposed item metadata and report status-scoped order namespaces; accepted/proposed order values may overlap without warning, while duplicate order values inside one accepted or proposed namespace are warning-level reviewability findings. Rewritten canonical item blocks omit empty strict path relationship fields such as `source_incubation`, `related_incubation`, `source_research`, `source_members`, `related_plan`, and `archived_plan`. It accepts advisory slice fields `--execution-slice`, `--slice-goal`, repeated `--slice-member`, repeated `--slice-dependency`, `--slice-closeout-boundary`, `--source-research`, repeated `--source-member`, repeated `--related-spec`, and repeated `--target-artifact` without making roadmap grouping lifecycle authority, and `--clear-field <field>` can clear reviewed first-class metadata on update without falling back to manual roadmap surgery. `plan --roadmap-item` may reuse the same item validation to refresh explicit source-incubation note relationship frontmatter back to the new active plan, and `plan`/`writeback` relationship sync may target existing compatibility RM sections while ignoring `slice_members` labels that are not roadmap ids. Targeted source-note relationship writes replace the full scalar block, including stale YAML list continuations, so an owned sync does not leave malformed frontmatter behind. It requires a readable `project/project-state.md` frontmatter and an existing regular roadmap file; `--action add` also requires canonical `## Items`. It refuses product-source fixtures, fallback/archive or generated-output roots, ambiguous roots, malformed roadmaps, duplicate ids, missing update targets, invalid statuses, invalid docs decisions, unsafe relationship paths, incompatible relationship destinations, missing related item ids, missing or malformed source-incubation frontmatter for reciprocal writes, self-relations outside slice membership, same-field set/clear conflicts, and symlink/path conflicts; and it does not infer priority, repair files, archive, stage, commit, approve closeout, or make roadmap output lifecycle approval.

  If the requested source-incubation note already carries owner-like relationship metadata, the roadmap item records the path as non-owning `related_incubation`, dry-run/apply reports that reused source-incubation posture, and the source note frontmatter is not retargeted.
- `meta-feedback --dry-run|--apply` composes the explicit `[MLH-Fix-Candidate]` incubation lane with managed canonical cluster metadata. Dry-run reports note body identity, dedupe posture, destination root, cluster route writes, explicit roadmap-detached posture, release boundary, and agent-operability guidance without writing files. If explicit flags are absent, structured note fields such as `signal_type`, `severity`, `affected_routes`, and `expected_owner_command` are used for the managed note and cluster metadata. `meta-feedback` does not create or update `project/roadmap.md`; mature clusters require explicit `roadmap --dry-run|--apply` promotion after review. `--to-root <central-root>` or `MYLITTLEHARNESS_META_FEEDBACK_ROOT` routes canonical writes to that central destination while `--root` remains observed-root provenance when `--from-root` is omitted; `--root` alone may receive canonical MLH product-debt capture only when it is already the central MyLittleHarness-dev root. If an ambient env destination overrides a central `--root`, the report must surface that precedence and show the exact `--to-root <central-root>` override to bypass stale workstation state. Refused meta-feedback dry-runs must render as preview refusals, not as reliable candidate/cluster previews with a future apply path. The command cannot approve repair, closeout, archive, lifecycle movement, next-plan opening, roadmap promotion, release removal, staging, commit, or push.

## Mutation Boundary

Mutating attach and repair behavior is gated by root classification before any filesystem write:

- A live operating root is an operator-provided target repository that owns operating memory and is not classified as product source, product compatibility fixture, legacy reference material, generated output, adapter state, cache, or archive-only material.
- A product-source compatibility fixture is any target whose state marks `root_role = "product-source"`, `fixture_status = "product-compatibility-fixture"`, or whose `product_source_root` resolves to the target root. Apply modes must refuse this target with exit code `2`; dry-run modes remain report-only and may return no-op proposals.
- A fallback, archive, generated-output, adapter, cache, log, local database, package archive, user config, PATH, hook, MCP, GitHub, browser, IDE, or workflow execution surface is outside the write boundary. Apply modes must refuse it with exit code `2`.

All allowed write paths must normalize under the explicit target root. A write that would escape the target root, follow ambiguous authority, rewrite an existing file, delete content, move content, archive content, commit changes, mutate user/workstation configuration, or infer authority from stale docs, weak semantic matches, generated projections, caches, adapters, or terminal output is forbidden.

The `projection` command is the manual refresh/recovery surface for generated artifacts and indexes after the initial attach-time build. Only build, rebuild, and delete modes may write or delete generated artifacts and the SQLite index under `.mylittleharness/generated/projection/`; inspect remains read-only. Those generated outputs cannot authorize attach, repair, archive, commit, plan lifecycle changes.

Read-only root classification is broader than mutation authority. Prose-style project-state assignments can make `status` and `validate` understand an operating root. The only mutating path from prose state is the selected `state-frontmatter-repair` class, which snapshots default-path `project/project-state.md`, prepends deterministic frontmatter, runs validation, and then stops before any other repair class.

Closeout/state writeback is a separate mutation boundary from repair. `writeback --apply` requires strict readable `project/project-state.md` frontmatter because it writes current closeout facts, lifecycle frontmatter, structured product-source metadata, and the derived Current Focus and Memory Routing Roadmap hot-pointer blocks rather than recovering missing state structure. It must update all MLH-owned derived active-plan copies it can safely identify during closeout synchronization, so stale `docs_decision: uncertain`, active-plan `status: active`, an active-phase body `- status: pending`, or stale managed hot-pointer prose after a completed phase is a diagnostic state, not the expected result of a successful closeout writeback. Explicit Docs Decision sections are derived closeout copies; schema examples and closeout checklist items are not derived copies. With `--from-active-plan`, exact active-plan closeout bullets are harvested as same-request closeout facts before preflight; when the active plan has no explicit closeout section, complete matching MLH closeout facts from `project/project-state.md` may be harvested as current authority. Explicit CLI closeout fields take precedence over harvested values. Closeout writeback records plan identity fields when available and reports whether prior facts are carried, replaced, or refused; partial closeout updates are refused when the recorded identity is absent or different from the current active or archived plan unless the same request supplies complete replacement facts. The only partial replacement exception is phase-only handoff with `docs_decision = uncertain` and `phase_status = complete`, which records the current plan identity and provisional docs decision without carrying stale prior facts. `writeback --apply --phase-status complete` writes the ready-for-closeout boundary without archiving, defaulting roadmap status, source-incubation archival, or next-slice movement. `writeback --apply --archive-active-plan` closes only the default active-plan lifecycle by moving the active plan into `project/archive/plans/` and clearing active pointers while preserving supplied or harvested facts. ## Repair Snapshot Contract

The MyLittleHarness repair subsystem owns the repair snapshot contract, but the operating root owns the snapshot files. A snapshot is safety evidence for a specific repair attempt, not workflow memory and not accepted truth.

Snapshot storage uses one target-bound boundary:

- Boundary: `.mylittleharness/snapshots/repair/` under the explicit target root.
- Snapshot directory name: `<YYYYMMDDTHHMMSSZ>-<repair-class>-<target-slug>-<hash-prefix>`.
- Required files: `snapshot.json` metadata plus byte-for-byte copies of pre-repair target files under `files/`.
- Path rule: every metadata path and copied file path must normalize under the explicit target root; symlink path segments, non-directory boundary conflicts, path traversal, absolute target paths inside metadata, and existing non-directory snapshot parents are refusal reasons.
- Product hygiene: snapshots must never be created in a product-source checkout or compatibility fixture. Product-root snapshots are debris, not fixtures.

Snapshot metadata must be inspectable and source-bound. The minimum metadata shape is:

- `schema_version`
- `created_at_utc`
- `tool_name` and `tool_version`
- `command`
- `root_kind`
- `repair_class`
- `target_root`
- `snapshot_root`
- `target_paths`
- `copied_files`
- `pre_repair_hashes`
- `planned_post_repair_paths`
- `source_diagnostics`
- `planned_route_entries`
- `retention`
- `rollback_instructions`
- `authority_note`

The `authority_note` must state that snapshots cannot approve repair, closeout, archive, commit, lifecycle decisions, or future repairs. A successful snapshot proves only that pre-repair bytes were copied and metadata was written before mutation started.

Retention is manual-first. MyLittleHarness must not silently delete, rotate, compress, move, or hide repair snapshots. Cleanup is an operator action after review/closeout, or a later explicit cleanup command with its own dry-run, refusal rules, and docs. Reports may warn about old or numerous snapshots, but warnings do not authorize deletion.

## Rollback And Idempotency

Create-only scaffold, AGENTS creation, docmap creation, and stable spec creation operations do not require backups because they never overwrite existing content. The implemented state frontmatter repair, lifecycle Markdown frontmatter repair, and docmap route repair require snapshots before rewriting existing files.

Any operation that would overwrite, delete, rename, normalize, resync, archive, or otherwise change existing content outside selected `project/project-state.md` frontmatter prepending, lifecycle Markdown frontmatter prepending, or `.agents/docmap.yaml` route-entry update is blocked unless a later implementation uses the repair snapshot contract above and starts with a no-write dry-run snapshot plan.

Rollback is manual-only for this contract. A create-only docmap repair must print instructions to remove the created docmap and only remove `.agents/` when the operator confirms it is empty and repair-created. A snapshot-protected repair must print rollback instructions that name the snapshot directory, copied files, target files, and validation command. No rollback command, cleanup command, archive action, VCS action, or lifecycle mutation is implied.

Allowed apply behavior must be idempotent: rerunning the same command against an already-created scaffold, already-created AGENTS contract, already-created docmap, already-created stable spec fixture, already-repaired state frontmatter, already-repaired lifecycle Markdown frontmatter, or already-repaired docmap returns success, reports no changed paths, and leaves file contents unchanged.
