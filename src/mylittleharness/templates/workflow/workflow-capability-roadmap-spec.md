# Workflow Capability Roadmap Spec

This stable workflow spec fixture anchors the expected capability roadmap surface for a MyLittleHarness operating root.

Repo-visible project files remain authoritative. This fixture is repair-created only when absent, and it cannot approve repair, closeout, archive, commit, rollback, lifecycle decisions, or future mutations.

This packaged stable-spec template is an intentionally minimal bootstrap stub: a repaired live root must treat it as review-required until human-maintained stable specs expand or replace it.

The optional live-root roadmap route is `project/roadmap.md`. It is a human-readable sequencing surface for accepted work between incubation and one active implementation plan; it is not part of the mandatory start path and cannot approve repair, closeout, archive, commit, rollback, or lifecycle decisions.

Roadmap item metadata should stay minimal and frontmatter-friendly: stable `id`, `status`, optional `stage`, advisory `order` or `priority`, `execution_slice`, `slice_goal`, `slice_members`, `slice_dependencies`, `slice_closeout_boundary`, `dependencies`, `source_incubation`, `related_incubation`, `source_research`, typed `source_members`, `related_specs`, `related_plan`, `archived_plan`, `target_artifacts`, `verification_summary`, `docs_decision`, `carry_forward`, `supersedes`, and `superseded_by`. Read-only diagnostics may validate these route relationships, accepted/active source-incubation evidence, accepted/active related-spec targets, status-scoped order namespaces, done items whose `docs_decision` is still `uncertain` while archived evidence is missing, and focused archive-context completeness when the file exists; `check --focus archive-context` reports archived-plan coverage, missing archive target root-cause classes with last repo-visible route traces and source-evidence posture, reconstructed source evidence, archived source-reference replacements under `project/archive/reference/**`, suspect incomplete-context archives, stale source references, and bounded recovery actions without writing files. The hidden `roadmap --dry-run|--apply` rail may add one explicit canonical item block, update one canonical item block or existing compatibility top-level `## RM-...` section including nonnumeric `RM-alpha`-style ids, refresh the MLH-owned Future Execution Slice Queue from accepted/proposed item metadata grouped by shared `execution_slice`, keep accepted/proposed order diagnostics separate, refresh an existing MLH-owned Current Posture section from active/accepted/proposed item metadata, preserve existing nonstandard scalar item fields, support explicit `stage`, safe repeated `--field key=value` custom scalar updates, repeated `--source-member <rel>` source evidence, repeated `--clear-field <field>` reviewed first-class metadata cleanup, omit empty strict path relationship fields from rewritten canonical item blocks, and compact older detailed canonical `done` item blocks with archived-plan evidence into Archived Completed History in an eligible live operating root. `source_members` is source/evidence membership, not reciprocal source ownership; it may point at draft, incubation, research, verification, or attachment routes, while generated `output/...` paths belong in `output_refs` on agent-run or verification evidence or in an imported route-owned artifact. `roadmap normalize --dry-run|--apply` may reorder only canonical managed item blocks into clear active, accepted, proposed, blocked/deferred, and terminal buckets, preserve each block body, refresh derived roadmap summaries, and avoid source-note relationship writes. Hidden `relationship-drift --dry-run|--apply` may scan one roadmap item or the whole roadmap, show before/after relationship graph rows, preview retarget/detach/source-metadata decisions, retarget stale active-plan `related_plan` edges to existing `archived_plan` routes, detach stale active-plan edges that no active plan owns, refresh reciprocal source metadata only for `source_incubation` and `related_incubation` routes, and report missing-route or missing-item impact; apply refuses unresolved impacts and cannot approve closeout, archive, roadmap promotion, lifecycle movement, staging, commit, push, rollback, or next-plan opening. When a requested source incubation is unowned, the rail may write reciprocal source-note relationship metadata; when it is already owned, it records non-owning `related_incubation` on the roadmap item without retargeting the source note. Hidden `incubate --apply` may record structurally known active-plan and active-plan roadmap relationship metadata in the target incubation note only when that note's topic or slug matches the current plan identity, primary roadmap item, execution slice, or covered roadmap item; unrelated recovery notes remain unclaimed. Hidden `incubation-reconcile --dry-run|--apply` may classify live incubation notes as active-roadmap-source, archived-covered, promoted-compacted, orphan-needs-triage, duplicate-or-superseded, or still-live-followup and apply only reconciliation metadata (`lifecycle_status`, `resolution`, `resolved_by`, `superseded_by`, `last_reconciled`) without deletion, archive, repair, roadmap mutation, staging, commit, or lifecycle approval. Hidden `meta-feedback --apply` may update configured central product-feedback canonical incubation cluster metadata without automatic roadmap placement; `--to-root <mlh-dev-root>` or `MYLITTLEHARNESS_META_FEEDBACK_ROOT` routes note and cluster writes to that central live operating root while the observed repo stays provenance, and `--root` alone may receive canonical MLH product debt only when it is already that central root. Exact-signature duplicates append automatically, and near-duplicates use explicit `--dedupe-to <canonical-id>`, which must name an existing canonical incubation note. Meta-feedback may read `signal_type`, `severity`, `affected_routes`, and `expected_owner_command` from the note body when flags are absent, reports that `project/roadmap.md` is left unchanged, and leaves mature-cluster promotion to explicit `roadmap --dry-run|--apply` review. Hidden `plan --roadmap-item` / `writeback --roadmap-item` may sync requested roadmap relationships plus active-plan or roadmap-slice `covered_roadmap_items` without making roadmap output lifecycle authority. Compatibility RM sections can keep work-area `slice_members` labels without expanding them as covered roadmap ids. Plan opening refreshes each explicit source-incubation note back to the new active plan; when a roadmap item carries only non-owning `related_incubation`, plan records that provenance in active-plan metadata and reports it without source-note writes. When `plan --update-active` replaces an active plan, the plan rail previews/applies a source-incubation relationship graph, retargets the new source note, and detaches the replaced source note's active-plan `related_plan` edge when that edge still points at `project/implementation-plan.md`. Plan synthesis split-boundary output may show archived-plan evidence for dependencies compacted from the live roadmap tail into Archived Completed History, and missing related-spec diagnostics should appear before a generated plan treats a stale spec path as docs/read/write scope. Targeted source-note writes replace the whole relationship scalar block, including stale YAML list continuations, as bounded relationship frontmatter block recovery rather than general repair. Reviewed transition archive skips the next roadmap item's source note before opening that next plan. If a writeback source incubation already lives under `project/archive/reference/incubation/**`, the relationship sync updates it in place and does not re-archive it.

Roadmap acceptance readiness treats provisional distilled research as a blocker when `source_research` or `source_members` point at `project/research/*.md` artifacts whose distillate quality gate records `planning_reliance="blocked"` or lacks enough gate coverage, source-bound claims, and confidence/uncertainty notes. That diagnostic is read-only and can only keep a roadmap item at `blocked-before-plan`; it cannot create research, synthesize judgment, mutate roadmap status, or approve lifecycle movement.

Roadmap dependencies may opt into an explicit slice-result gate with custom metadata such as `slice_result_gate="decision-packet"` and an optional `slice_result_artifact` path. When a downstream `plan --roadmap-item` depends on that item, plan opening must refuse until the upstream item is terminal `done` and a repo-visible decision packet records `safe_to_continue_existing_sequence: true`. If the packet records `new_slice_candidates`, `scope_expansions`, `blocked_followups`, `fork_decision`, or `roadmap_updates_required`, the downstream item stays blocked until the operator updates or adds roadmap slices through an explicit `roadmap --dry-run|--apply` review. The gate is sequencing evidence only; it does not approve archive, roadmap promotion, lifecycle movement, staging, commit, push, or source mutation.

Plan, writeback, roadmap, and transition route writes must preview and verify newly required route references in the same transaction. Required `active_plan`, `last_archived_plan`, and terminal roadmap `archived_plan` pointers are valid only when the target is safe, root-relative, concrete, and either already present or created by the same bounded apply; otherwise dry-run reports the unresolved reference and apply refuses before writing.

Relationship metadata stays path/id based: source incubation/research, related roadmap item, active or archived plan, specs, decisions, ADRs, verification, archive, supersession, merge, split, and rejection links are readable Markdown/frontmatter fields, not a separate authoritative index. Generated relationship graphs such as `.mylittleharness/generated/projection/relationships.json` are rebuildable navigation projections only. Whole-file incubation auto-archive is safe only for structurally covered notes: single-entry notes can be covered by the closeout chain, while mixed notes require terminal `## Entry Coverage` bullets for every dated entry. Mixed or open-thread notes without that coverage remain active and receive blocker diagnostics plus heuristic split suggestions. Implemented incubation note archives use `project/archive/reference/incubation/**`.
