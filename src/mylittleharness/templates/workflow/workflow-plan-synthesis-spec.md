# Workflow Plan Synthesis Spec

This stable workflow spec fixture anchors the expected plan synthesis surface for a MyLittleHarness operating root.

Repo-visible project files remain authoritative. This fixture is repair-created only when absent, and it cannot approve repair, closeout, archive, commit, rollback, lifecycle decisions, or future mutations.

Roadmap-backed plans should keep the executable one-slice contract in active-plan frontmatter: `primary_roadmap_item`, `covered_roadmap_items`, `domain_context`, `target_artifacts`, `execution_policy`, `auto_continue`, `stop_conditions`, and `closeout_boundary`. `plan --roadmap-item` may treat only `accepted` or `active` roadmap items as current plan-opening inputs; proposed, deferred, rejected, done, superseded, blocked, missing-status, or historical roadmap facts must refuse or hard-warn before active-plan, project-state lifecycle, roadmap relationship, or source-incubation relationship write previews are emitted. When roadmap `target_artifacts` are empty but source incubation preserves `affected_routes` or explicit source/test/docs path mentions, plan synthesis may promote exact source/test/docs targets into `target_artifacts` while leaving lifecycle-only routes and broad directory or wildcard hints as context. For accepted or active roadmap items, stale `slice_closeout_boundary` wording that still says no implementation plan, no archive, no lifecycle movement, or provisional pre-implementation attachment should be normalized into a non-authority safety note and reported before the generated plan is relied on for closeout boundaries. The default generated policy is current-phase-only with `auto_continue = false`; verification success does not authorize a later phase by itself. `project/project-state.md` remains lifecycle pointer authority rather than slice-membership storage.

When a roadmap-backed plan comes from a meta-feedback incubation cluster, source excerpt selection should prefer the tagged `[MLH-Fix-Candidate]` entry over provenance or cluster boilerplate, preserve the full cleaned representative observation, and carry route/owner hints such as `affected_routes` and `expected_owner_command` into generated task context. When multiple tagged entries exist, selection prefers the latest route-rich non-recovery entry, then the latest route-rich entry, then the latest tagged entry. If that richer source excerpt has route hints, is not recovery-only evidence, and the roadmap item is not part of a grouped execution slice, it may become the plan `domain_context` and derived objective instead of a shorter roadmap `slice_goal`. Recovery-only source notes and grouped accepted slices should preserve roadmap `slice_goal` or `carry_forward` as the leading plan objective/task context.

Roadmap-backed plan synthesis may report bundle rationale, split boundary, `target_artifact_pressure`, and `phase_pressure` in dry-run/apply output and generated `Plan Synthesis Notes`. These are advisory sizing signals only; they cannot approve repair, closeout, archive, commit, rollback, lifecycle decisions, or future mutations.

Research distillation is a validation rail, not a synthesis authority. A distilled research artifact should record `quality_status`, `planning_reliance`, gate-question coverage, source-bound claims, and confidence/uncertainty notes. Shallow distilled artifacts that lack those fields or sections remain provisional and must not be treated as enough `source_research` evidence for roadmap-backed plan opening.

When roadmap metadata carries enough target-artifact, verification, spec, or docs-update pressure, generated plans should include a `Phase Outline` and concrete phase sections with `write_scope`, `verification_gates`, docs-decision handling, and state-transfer expectations. Low-pressure roadmap plans may stay one current phase only when `Plan Synthesis Notes` records an explicit one-shot rationale and stop condition for scope expansion. When a source implementation phase is verified by target test artifacts, those adjacent regression tests should be named in that phase's write scope. When roadmap `docs_decision` is `updated`, synthesis should report `plan-docs-write-scope-impact` and name docs/spec/package metadata files in the docs phase, or require exact docs scope before mutation when the roadmap omitted it. Directory-level docs/spec hints such as `docs/specs` are broad context, not exact write scope. Missing roadmap `related_specs` targets should be reported by `check` and `plan --roadmap-item` before generated plans rely on them as read context or docs/write scope; operators should retarget/remove stale specs or keep `docs_decision='uncertain'` rather than inventing absent stable specs.

Generated `verification_gates` should be evidence-backed by target test artifacts or repo-visible command discovery from package scripts, Makefile/just/task files, CI workflow commands, documented commands, or roadmap verification summaries. When that evidence is weak or absent, the gate should stay `UNRESOLVED` instead of falling back to a universal toolchain command. Broad pytest/full-suite gates over source-only target artifacts should surface adjacent regression-test ownership until tests are named or the gate is narrowed.

Read-only grain diagnostics may inspect active plans for missing slice metadata, target-artifact gaps, vague write scope, generic or unresolved verification, toolchain-mismatched verification gates, adjacent regression-test ownership gaps, over-atomic slices, giant brittle slices, and unsafe auto-continuation posture. They are calibration inputs only and do not rewrite plans or promote numeric thresholds into hard gates.
